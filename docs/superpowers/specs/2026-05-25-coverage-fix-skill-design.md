# Design: `/n-coverage-fix` skill + COVERAGE.md recommendations

**Date:** 2026-05-25  
**Status:** approved  
**Project:** mlmail + @nitra/cursor

---

## Problem

1. **Stability:** `bun coverage` crashes mid-run (SIGURG, macOS memory pressure) — progress lost, run must restart from zero.
2. **No actionable output:** `COVERAGE.md` shows aggregate scores but doesn't guide the developer (or an LLM agent) on which tests to add.
3. **Manual test-writing:** After seeing low mutation score (e.g., 67.61% in mlmail JS), there's no automated path to improve it.

---

## Goals

1. Stryker resumes from checkpoint after crash — no full re-run needed.
2. `COVERAGE.md` gains a `## Recommendations` section: file-level, line-level mutations that survived, with an example existing test to establish code style.
3. New skill `/n-coverage-fix` runs `bun coverage`, reads recommendations, writes missing tests, and iterates until mutation score stops improving (convergence).

---

## Non-Goals

- Re-implementing Stryker's mutation selection logic.
- Targeting 100% mutation score — convergence stops when no improvement, not at a fixed threshold.
- Supporting non-JS/non-Rust coverage providers (Rust already at 100%).

---

## Design

### 1. Stability: Stryker incremental mode

**Change in `@nitra/cursor`:** add to `rules/test/js/data/stryker_config/stryker.config.baseline.mjs`:

```js
incremental: true,
incrementalFile: 'reports/stryker/incremental.json',
```

Stryker writes each mutant result to the incremental file as it completes. On the next run, already-tested mutants are skipped.

**`.gitignore`:** already covered by existing pattern `**/reports/stryker/` — no change needed.

**Existing projects (e.g., mlmail):** baseline copy only happens when `stryker.config.mjs` doesn't exist. For mlmail, add `incremental` to `app/stryker.config.mjs` manually (it has custom `mutate` patterns).

---

### 2. `## Recommendations` section in COVERAGE.md

**Where:** `rules/js-lint/coverage/coverage.mjs` — after `parseStrykerReport()`, also extract survived mutants grouped by file.

**For each survived mutant group, the section includes:**

````markdown
## Recommendations

> Автоматично генерується після `bun coverage`. Використовується `/n-coverage-fix`.

### src/i18n/auth-errors.js

**Вижило мутантів: 8**

| Рядок | Тип                   | Оригінал                              | Вижив                                   |
| ----- | --------------------- | ------------------------------------- | --------------------------------------- |
| 19    | ConditionalExpression | `kind !== null && kind !== undefined` | `false`                                 |
| 19    | LogicalOperator       | `kind !== null && kind !== undefined` | `kind === null \|\| kind === undefined` |

**Приклад наявного тесту** (з `src/i18n/auth-errors.test.js`):

```js
it('повертає код для INVALID_EMAIL', () => {
  expect(authError({ kind: 'INVALID_EMAIL' })).toBe('auth.errors.invalidEmail')
})
```
````

**Що треба протестувати:** умову на рядку 19 — передай `null` і `undefined` окремо як `kind`, перевір що результат відрізняється від valid-kind, щоб `&&` не замінився на `||` непоміченим.

```

**How the example test is found:** the provider resolves the test file path from the mutated source file path (e.g., `src/foo.js` → `src/foo.test.js` or `src/tests/foo.test.js`), reads the first `it(` or `test(` block as the style example.

**"Що треба протестувати" text:** generated by an LLM API call from within `coverage.mjs` (Bun/Node runtime). Input: mutated source lines ± context (~10 lines), mutation type, original vs survived replacement. Output: 2-3 sentences in Ukrainian describing what test scenario would catch this mutation. Uses `@anthropic-ai/sdk` (or raw `fetch` to Anthropic API) with `ANTHROPIC_API_KEY` env var. If key is not set → field is omitted (recommendations still written, but without the "Що треба протестувати" paragraph). Prompt caching used if multiple mutants from the same file (system prompt cached).

### 3. `/n-coverage-fix` skill algorithm

```

1. Запусти `bun coverage` — чекай завершення
2. Прочитай COVERAGE.md секцію ## Recommendations
3. Якщо секції немає або порожня → DONE (нема вижилих мутантів)
4. Запам'ятай поточний mutation score як baseline_score
5. Для кожного файлу у Recommendations:
   а. Прочитай source-файл (mutated lines ± context)
   б. Прочитай відповідний test-файл
   в. Використай приклад тесту з Recommendations як style guide
   г. Напиши нові тести що покривають кожен вижилий мутант
6. Запусти `bun test`
   → ЯКЩО FAIL:
   - Не відкочувати зміни
   - Показати user: яка помилка, які файли змінені, що вже покращено
   - Очікувати рішення: [виправити вручну → продовжити] / [пропустити файл] / [зупинити]
7. Запусти `bun coverage` знову
   → ЯКЩО CRASH: нагадати user що Stryker incremental зберіг прогрес → перезапустити `bun coverage`
8. Порівняй новий mutation score з baseline_score
9. Якщо НЕ покращився → DONE (конвергенція)
10. Інакше: baseline_score = новий score → крок 5 (наступна ітерація)

```

**Error handling принцип:** checkpoint-based. Кожна успішна ітерація фіксована в тестовому файлі та COVERAGE.md. Помилка = пауза для людини з повним контекстом, не скасування прогресу.
```
