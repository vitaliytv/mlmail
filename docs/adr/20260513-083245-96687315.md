## ADR Виправлення Google OAuth client_id: runtime замість compile-time

**Контекст:** Під час першого тестового логіну на macOS застосунок повертав `401 invalid_client` — Google не знаходив OAuth client. Причина: `option_env!()` читає значення env-змінної під час компіляції, тому при `cargo build` без активного env у shell у бінарнику осідав буквальний fallback-рядок `REPLACE_ME_DESKTOP_CLIENT_ID`.
**Рішення/Процедура/Факт:** Додано crate `dotenvy v0.15.7`. `auth/config.rs` переписано: три функції `desktop_client_id()`, `android_client_id()`, `android_web_client_id()` через `std::env::var()` у `OnceLock<String>`, із викликом `dotenvy::dotenv().ok()` у `lib.rs::run()` до `tauri::Builder`. `option_env!()` з compile-time-fallback повністю прибрано. Споживачі в `auth/mod.rs` перейшли з констант на виклики функцій. Виконано `cargo clean -p mlmail` та rebuild.
**Обґрунтування:** `option_env!()` — правильний інструмент для токенів, якими керує CI/CD build pipeline; але для інтерактивної розробки він порушує workflow (треба `source .env` перед кожним `cargo build`). Runtime-читання через `dotenvy` перетворює `.env` на гарячу конфігурацію: зміна файлу підхоплюється наступним запуском процесу без перекомпіляції.
**Розглянуті альтернативи:** Залишити `option_env!()` з обов'язковим `set -a; source .env; set +a` перед build (відхилено як крихке); передавати client_id через Tauri plugin config в `tauri.conf.json` (відхилено: немає шифрування, дублювання файлу).
**Зачіпає:** `app/src-tauri/src/auth/config.rs`, `app/src-tauri/src/auth/mod.rs`, `app/src-tauri/src/lib.rs`, `app/src-tauri/Cargo.toml`, `app/src-tauri/.env`

## ADR Зберігання app/src-tauri/.env у приватному репо

**Контекст:** `.env` з трьома Google OAuth Client ID спочатку додано до `.gitignore`. Розробник запитав, чи можна прибрати його з ignore-списку, щоб Client ID не треба було копіювати вручну.
**Рішення/Процедура/Факт:** Рядок `app/src-tauri/.env` прибрано з `.gitignore`; файл `.env` закомічено в приватний репозиторій `vitaliytv/mlmail` (GitHub, visibility: private).
**Обґрунтування:** Google OAuth Client ID для native desktop/mobile-застосунку є **публічними ідентифікаторами**: вони з'являються у мережевих запитах під час будь-якого логіну і тривіально витягуються з дистрибутивного бінарника командою `strings`. Google прямо розмежовує "Client ID" (public) та "Client secret" (private, релевантний лише для backend/web-apps із server-side token exchange). У `.env` немає `client_secret` — лише три Client ID. Ризик витоку незначний, зручність — пряма.
**Розглянуті альтернативи:** Тримати `.env` в `.gitignore` і шерити ID поза репо (відхилено: незручно і не додає реальної безпеки); зашифрувати `.env` через `git-crypt` (відхилено: overhead непропорційний до ризику для private native-app ID).
**Зачіпає:** `.gitignore`, `app/src-tauri/.env`, `app/src-tauri/.env.example`

## ADR Google OAuth Internal brand через IAP для домену nitralabs.com

**Контекст:** MLMaiL потребував OAuth consent screen для Google OAuth. Зовнішній тип вимагає проходження верифікації Google для sensitive scopes (`gmail.modify`). Проєкт `vitaliytv` знаходиться всередині Workspace-організації `nitralabs.com`.
**Рішення/Процедура/Факт:** OAuth consent brand створено як **Internal** (`orgInternalOnly: true`) через `gcloud iap oauth-brands create`, що автоматично обмежує логін лише акаунтами `*@nitralabs.com`. Scopes (`openid`, `userinfo.email`, `gmail.modify`) додаються через UI Google Auth Platform → Data Access. Три OAuth Client ID (Desktop, Android, Web) все одно створюються вручну через UI `console.cloud.google.com/apis/credentials`.
**Обґрунтування:** Internal-тип усуває необхідність верифікації Google для sensitive scopes і знімає 100-user cap. Для продукту на початкових етапах, де аудиторія — тільки команда `nitralabs.com`, це оптимальний шлях з нульовою процедурною затримкою. Перехід на External + verification — явна майбутня дія при масштабуванні.
**Розглянуті альтернативи:** External тип з Testing-mode (cap 100 users) — відхилено через незручність управління test users; External тип з повним verification (до 30 днів) — відхилено як передчасне.
**Зачіпає:** Google Cloud Console `vitaliytv`, OAuth consent screen, `app/src-tauri/.env`
