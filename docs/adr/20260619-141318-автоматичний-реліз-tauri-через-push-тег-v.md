## ADR Виключення `Swatinem/rust-cache` з release workflow

## Context and Problem Statement
При першому прогоні `bun run lint-ga` zizmor повідомив про cache-poisoning ризик у контексті release-workflow через використання `Swatinem/rust-cache@v2`.

## Considered Options
* Прибрати `rust-cache` з `release.yml`
* Залишити `rust-cache` і задокументувати suppression у `zizmor.yml`

## Decision Outcome
Chosen option: "Прибрати `rust-cache` з `release.yml`", because релізи рідкісні і кешування не критичне для них, а security-рейтинг не варто знижувати без потреби.

### Consequences
* Good, because `release.yml` пройшов zizmor з `No findings to report` після видалення.
* Bad, because cold-build Android займатиме більше часу (повне перекомпілювання Rust-залежностей).

## More Information
Команда перевірки: `bun run lint-ga 2>&1 | grep -iE "release.yml|finding"`. Zizmor-конфіг: `.github/zizmor.yml`.

## ADR Окремий keystore на кожен застосунок

## Context and Problem Statement
Три застосунки (mlmail, myshare, task) є окремими продуктами в різних GitHub-організаціях; потрібно вирішити, чи використовувати один спільний Android keystore, чи окремі.

## Considered Options
* Окремий keystore на кожен застосунок
* Один спільний keystore для всіх трьох

## Decision Outcome
Chosen option: "Окремий keystore на кожен застосунок", because різні Play-лістинги вимагають незалежних ключів підпису; компрометація одного ключа не впливає на інші застосунки.

### Consequences
* Good, because ізоляція ключів: кожен app має свій `ANDROID_KEY_ALIAS` (`mlmail`, `myshare`).
* Bad, because більше keystores для зберігання та бекапу; `~/keystores-release/mlmail/` і `~/keystores-release/myshare/` — критично важливі файли, втрата яких унеможливлює оновлення в Google Play.

## More Information
Keystores згенеровані у `~/keystores-release/<app>/`, поза git-репозиторіями; усі значення секретів у `~/keystores-release/<app>/secrets/`. Команда верифікації: `keytool -list -keystore ~/keystores-release/mlmail/mlmail-release.jks -storepass <password>`.
