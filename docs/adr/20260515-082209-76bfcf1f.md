## ADR Команди Gmail через Rust, не через fetch у Vue

**Контекст:** MLMaiL потребує звернень до Gmail REST API (inbox count, випадковий лист). Виникло питання, де виконувати HTTP-запити — у Rust-бекенді Tauri чи напряму у Vue через `fetch`.

**Рішення/Процедура/Факт:** Усі Gmail API-виклики реалізовані як Tauri-команди у Rust (новий модуль `app/src-tauri/src/gmail/`). Vue-сторона викликає лише `invoke('gmail_inbox_count')` та `invoke('gmail_random_message')` — жодного `fetch` до googleapis у JS-коді.

**Обґрунтування:** Access token не виходить за межі Rust (не потрапляє у JS-контекст WebView); відсутні CORS-проблеми у Tauri WebView; логіка рефрешу токена (`acquire_access_token`) вже є в `auth/mod.rs` і може бути перевикористана напряму; легше тестувати через mockito (Rust) ніж mocking fetch (JS).

**Розглянуті альтернативи:** Прямий `fetch` з Vue до Gmail API — відкиднуто через ризик витоку токена у JS і CORS-непередбачуваність у WebView.

**Зачіпає:** `app/src-tauri/src/gmail/mod.rs`, `app/src-tauri/src/gmail/error.rs`, `app/src-tauri/src/auth/mod.rs` (helper `acquire_access_token`), `app/src/services/auth-store.js`

## ADR Endpoint для кількості листів INBOX — `users.labels.get?id=INBOX`

**Контекст:** Стартовий екран MLMaiL показує кількість усіх листів у мітці INBOX. Потрібно визначити, який Gmail API endpoint дає точне число одним викликом.

**Рішення/Процедура/Факт:** Команда `gmail_inbox_count` викликає `GET https://gmail.googleapis.com/gmail/v1/users/me/labels/INBOX` і повертає поле `messagesTotal` (тип `u64`).

**Обґрунтування:** Один HTTP-виклик, точне число (не estimate), канонічний шлях для підрахунку по конкретній мітці. `users.getProfile` повертає загальну кількість по всій скриньці (включно з архівом) — не те, що потрібно. `messages.list?maxResults=1` дає лише `resultSizeEstimate` — приблизне значення.

**Розглянуті альтернативи:** `users.getProfile` (загальний count, не INBOX-специфічний); `messages.list?labelIds=INBOX` з `resultSizeEstimate` (наближене, не точне).

**Зачіпає:** `app/src-tauri/src/gmail/mod.rs:fetch_inbox_count`, `app/src-tauri/src/gmail/mod.rs` (struct `LabelResponse { messages_total }`)

## ADR Inbox count і random message зберігаються в `auth-store`, не в окремому composable

**Контекст:** Gmail-дані (inbox count, поточний лист) потрібні у `Login.vue`. Постало питання: виносити їх у окремий `mailbox-store.js` (як спочатку накидали хуки) чи тримати в `auth-store.js`.

**Рішення/Процедура/Факт:** `inboxCount`, `inboxErrorKind`, `currentMessage`, `messageErrorKind`, `isMessageLoading` та методи `refreshInboxCount()`, `loadRandomMessage()` додані безпосередньо до `auth-store.js`. Окремий `mailbox-store.js` створений хуком, але видалений як непотрібний.

**Обґрунтування:** Gmail-дані на стартовому екрані семантично прив'язані до auth-сесії: вони фетчаться після `login()`/`initialize()` і чистяться після `logout()`. Виносити в окремий store — передчасна абстракція (YAGNI); жодних інших споживачів mailbox-стору немає. Одна залежність простіша, ніж дві скоординовані.

**Розглянуті альтернативи:** Окремий `mailbox-store.js` — відкинутий; він дублює lifecycle-логіку auth-store і не має додаткових споживачів.

**Зачіпає:** `app/src/services/auth-store.js`, `app/src/services/auth-store.test.js`, `app/src/views/Login.vue`

## ADR Вибірка випадкового листа з перших 100 повідомлень INBOX

**Контекст:** Для показу випадкового листа на стартовому екрані потрібно визначити «простір вибірки»: скільки листів завантажувати, щоб обрати один.

**Рішення/Процедура/Факт:** Команда `gmail_random_message` спочатку викликає `messages.list?labelIds=INBOX&maxResults=100&fields=messages/id`, отримує масив id, обирає випадковий індекс через `rand`, потім робить `messages.get?format=full` для обраного id. Body: перший `text/plain`-парт (base64url-decoded), fallback — `text/html` зі стрипом тегів; обмеження 10K символів.

**Обґрунтування:** Два HTTP-виклики, передбачувана швидкість, без додаткових квот. Чесна рівномірна вибірка по всьому INBOX потребувала б пагінацію — для скриньки в 5K листів це 50+ послідовних викликів, що неприйнятно для UX стартового екрану. «Випадковість серед останніх 100» достатня для цілі «показати щось цікаве».

**Розглянуті альтернативи:** Повна пагінація INBOX для рівномірної випадковості — відкинута через надмірну кількість викликів; рандомізація за датою через `q=before/after` — складна і схильна до помилок округлення.

**Зачіпає:** `app/src-tauri/src/gmail/mod.rs` (нова команда `gmail_random_message`), `app/src-tauri/src/gmail/error.rs` (новий kind `Empty`), `app/src/services/auth-store.js`, `app/src/views/Login.vue`, `docs/superpowers/specs/2026-05-15-random-message-design.md`
