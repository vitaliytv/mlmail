## ADR Вибір Quasar як UI-кіт для Tauri macOS + Android

**Контекст:** MLMaiL — Tauri-застосунок, що запускається на macOS і Android. Правило `.cursor/rules/vue.mdc` вже передбачає Quasar. Після реалізації inbox-count і random-message-фічі виник момент переосмислення UI-підходу (Login.vue написаний на plain HTML).

**Рішення/Процедура/Факт:** Обрано Quasar як єдину UI-бібліотеку для Vue-компонентів MLMaiL. Plain HTML і Reka UI відхилені на користь Quasar. Наступний крок — окрема ітерація: `@quasar/vite-plugin` у `vite.config.js`, `Quasar` plugin у `main.js`, рефакторинг `Login.vue` на `<q-btn>`, `<q-card>`, `<q-banner>`, `<q-chip>`, `<q-spinner-dots>`.

**Обґрунтування:** (1) Material Design — рідний UX-стандарт Android, де очікуються ripple, bottom-sheet-діалоги, swipe-to-dismiss, safe-area; (2) `Platform.is.mac` / `is.android` дозволяє один компонент з дрібними дельтами замість двох кодових гілок; (3) bundle size не критичний — Tauri-бінарник ~10 МБ, Quasar +120 КБ = 1%; (4) майбутні фічі MLMaiL (список листів, діалоги видалення, `q-pull-to-refresh`, `q-virtual-scroll`, `q-skeleton`) вже є в Quasar і не потребують написання з нуля.

**Розглянуті альтернативи:**
- **Plain HTML + scoped CSS** — відхилено: жодних готових компонентів для mobile UX-патернів, усе писати вручну.
- **Reka UI (Radix Vue)** — headless-примітиви без дизайну; підходить для власної design-system з Tailwind/shadcn-vue; відхилено як overkill для поточної команди; залишається опцією якщо з'явиться дизайнер із власною design-system.

**Зачіпає:** `app/src/main.js`, `app/vite.config.js`, `app/src/views/Login.vue`, `app/package.json`, `.cursor/rules/vue.mdc`; непрямо — всі майбутні `.vue`-файли.

## ADR Показ випадкового листа INBOX на стартовому екрані

**Контекст:** Після реалізації inbox-count потрібно додати предметний контент на стартовий екран: показати один довільний лист з INBOX (від/тема/дата + plain-text тіло) із кнопкою «Показати інший».

**Рішення/Процедура/Факт:** Нова Tauri-команда `gmail_random_message` у `app/src-tauri/src/gmail/mod.rs`: (1) `GET messages?labelIds=INBOX&maxResults=100&fields=messages/id` → масив id; (2) `rand::thread_rng().gen_range(0..n)` → випадковий індекс; (3) `GET messages/<id>?format=full` → парсинг `payload.headers` (From, Subject, Date) і рекурсивний обхід `payload.parts` для тіла. Новий файл `gmail/message.rs` з `GmailMessage { id, from, subject, date, body }` і хелперами `extract_header` та `extract_plain_text`. Auth Store отримав `currentMessage`, `messageErrorKind`, `isMessageLoading`, `loadRandomMessage()`. `Login.vue` рендерить картку з `<pre>` для тіла і кнопку «Показати інший».

**Обґрунтування:** (1) Sample-space «перші 100» (`maxResults=100`) — два HTTP-виклики незалежно від розміру скриньки; (2) `text/plain` перед `text/html` (html→text через regex + `html-escape`) — мінімум залежностей, достатня якість для preview; (3) тіло обрізається до 10 000 символів, щоб не перевантажувати UI при листах-гігантах; (4) `GmailError::Empty` додано для порожніх скриньок із власним i18n-рядком.

**Розглянуті альтернативи:**
- **Повна пагінація всього INBOX** для справжньої рівномірної вибірки — відхилено: 50+ послідовних запитів для скриньок з 5K+ листів, overkill для UX-фічі preview.
- **Тільки snippet (без повного text/plain)** — простіший парсинг, але Gmail сам обрізає snippet до ~200 символів без збереження структури абзаців.

**Зачіпає:** `app/src-tauri/src/gmail/mod.rs`, `app/src-tauri/src/gmail/message.rs` (new), `app/src-tauri/src/gmail/error.rs`, `app/src-tauri/Cargo.toml` (+`regex`, `html-escape`), `app/src/services/auth-store.js`, `app/src/views/Login.vue`, `app/src/i18n/auth-errors.js`, `docs/ci4/03-components.md`, `docs/ci4/04-code.md`.
