## ADR Міграція тестового фреймворку з bun:test на Vitest

## Context and Problem Statement

Синк `npx @nitra/cursor` підняв пакет до v11.1.0 і додав `app/vitest.config.mjs` та `scripts/vitest.config.mjs`. Проєкт використовував `bun test` з ручним `test/happy-dom.preload.js` та `@happy-dom/global-registrator` для реєстрації DOM і мокування імпортів. Через конфлікт між bun:test-сетапом і v11-шаблонами linter (`knip`, `jscpd`) падав; автоматично регенеровані конфіги щоразу перезаписувались Stop-хуком.

## Considered Options

* Залишити `bun test` і заглушити lint-errors точковими ignore-правилами
* Мігрувати на Vitest — привести реалізацію у відповідність до v11-шаблонів і `nitra/task`-канону

## Decision Outcome

Chosen option: "Мігрувати на Vitest", because користувач прямо вказав «перемігровуємо на vitest» і це усуває root-cause регенерації v11-конфіктів, яка брудила дерево щоходу.

### Consequences

* Good, because `vitest.config.mjs` через `mergeConfig(vite.config.js, …)` використовує наявні Vite-плагіни (Vue, VueMacros, AutoImport, Quasar), тож SFC-компіляція і авто-імпорти в тестах нативні — `happy-dom.preload.js` і `@happy-dom/global-registrator` видалено.
* Good, because `stryker.config.mjs` перейшов на `testRunner: 'vitest'` з `perTest`-режимом (швидший mutation score за рахунок часткового ре-тесту).
* Good, because Stop-хук більше не перетирає конфіги деструктивно — v11-шаблони тепер коректні й використовуються.
* Bad, because трюк `?initial-state` (query-suffix для свіжого модуля) не підтримується Vitest — замінено на `vi.resetModules()`; `mock.module` з `bun:test` замінено на `vi.hoisted`+`vi.mock`.

## More Information

Змінені файли: `app/vitest.config.mjs`, `app/stryker.config.mjs`, `app/src/**/*.test.js` (5 файлів), `app/package.json` (scripts + deps), root `package.json` (+`vitest`, `@vitest/coverage-v8`, `@stryker-mutator/vitest-runner`), `scripts/package.json` (+`vitest`). Видалено: `app/test/happy-dom.preload.js`, deps `@happy-dom/global-registrator`, `@types/bun`, `@vue/compiler-sfc`. Усі 85 тестів пройшли під vitest перед комітом.

## ADR Вбудований Desktop OAuth client_secret у вихідний код

## Context and Problem Statement

macOS-логін завершувався помилкою «Помилка авторизації Google.» (`AuthError::OAuth`). Файл `.env.secret` був відсутній, а `desktop_client_secret()` повертав рядок-placeholder `REPLACE_ME_…` — Google відхиляв token-exchange запит з `invalid_client`. Варіант зберігати секрет виключно у `.env.secret` вимагав від кожного розробника ручного налаштування для будь-якого запуску.

## Considered Options

* Зберігати `MLMAIL_GOOGLE_DESKTOP_CLIENT_SECRET` лише в `.env.secret` (gitignored)
* Вшити `client_secret` у вихідний код (`config.rs`) і додати виняток для trufflehog

## Decision Outcome

Chosen option: "Вшити client_secret у вихідний код", because користувач явно попросив це зробити, вказавши, що Desktop-клієнт аналогічний інсталованому застосунку — секрет однаково присутній у кожному бінарнику.

### Consequences

* Good, because transcript фіксує очікувану користь: логін запрацював одразу після перезбірки без жодних зовнішніх конфіг-файлів.
* Good, because `desktop_client_secret()` залишає env-override: якщо `MLMAIL_GOOGLE_DESKTOP_CLIENT_SECRET` задано у `.env.secret` — використовується він (backward-compatible).
* Bad, because секрет присутній у git-історії та у тексті цього чату; компрометація git-репозиторію або витік чату розкривають його. trufflehog-scan виключає `app/src-tauri/src/auth/config.rs$` — якщо виняток коли-небудь прибрати, CI одразу спіймає.

## More Information

Змінені файли: `app/src-tauri/src/auth/config.rs` (константа `DESKTOP_SECRET_BUILTIN`, оновлена `desktop_client_secret()`), `.trufflehog-exclude` (новий рядок `app/src-tauri/src/auth/config\.rs$`). Примітка для майбутнього: Google розглядає `client_secret` Desktop-клієнта як «не конфіденційний» ([OAuth 2.0 for Mobile & Desktop Apps](https://developers.google.com/identity/protocols/oauth2/native-app)) — реальний захист флоу забезпечує PKCE, а не секрет.

## ADR Ранній гард `ConfigMissing` для ненастроєних OAuth credentials

## Context and Problem Statement

При порожньому або placeholder `client_id`/`client_secret` функція `run_login` запускала повний OAuth-флоу, отримувала від Google `invalid_client` і відображала користувачу невиразне «Помилка авторизації Google.» — без підказки, що саме не налаштовано. Перевірка `is_real_client_id` не відхиляла порожній рядок.

## Considered Options

* Залишити поточну поведінку (помилка маскується під OAuth-помилку)
* Додати новий варіант помилки `ConfigMissing` і ранній гард у `run_login` перед стартом флоу

## Decision Outcome

Chosen option: "Додати `ConfigMissing` і ранній гард", because Claude запропонував, користувач підтвердив («Зробити цей гард»).

### Consequences

* Good, because transcript фіксує очікувану користь: замість «Помилка авторизації Google.» користувач бачить «Google OAuth не налаштовано: заповніть credentials у .env / .env.secret.» з іменем конкретної env-змінної.
* Good, because `is_real_client_id` тепер відхиляє порожній і пробільний рядок (`!value.trim().is_empty() && !value.starts_with(PLACEHOLDER)`), що покриває новий тест `empty_or_blank_value_is_not_considered_real`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

Змінені файли: `app/src-tauri/src/auth/error.rs` (новий варіант `ConfigMissing(String)`), `app/src-tauri/src/auth/config.rs` (оновлена `is_real_client_id`, нова функція `require_configured`, нові Rust-тести), `app/src-tauri/src/auth/mod.rs` (`run_login` macOS та Android з ранньою перевіркою через `require_configured`), `app/src/i18n/auth-errors.js` (повідомлення `ConfigMissing`), `app/src/i18n/auth-errors.test.js` (новий JS-тест). Rust config-тести: 3/3. JS-тести: 86/86.
