## ADR Стратегія e2e-тестування Tauri-додатку MLMaiL

**Контекст:** Потрібно покрити e2e-сценарії Tauri v2-додатку (Vue 3 + Quasar, Rust-бекенд з Google OAuth + Gmail API). macOS — основна платформа розробки, але `tauri-driver` не підтримує WebKit WebDriver на macOS; OAuth-флоу неможливо пройти в WebDriver-сесії через реальний Google без моків.

**Рішення/Процедура/Факт:** Обрано варіант «справжній e2e через WebDriverIO»: реальний бінарник Tauri, WebDriver (`webkit2gtk-driver` + `tauri-driver`), ручний запуск (без CI). Google OAuth і Gmail API мокуються через локальний HTTP-стаб (наприклад, `mockito` або окремий сервер); для цього `BASE_URL` Google endpoints виноситься з хардкоду в Rust у конфіг/env var. Запуск — виключно на Linux (або Linux VM), оскільки macOS не має WebKit WebDriver.

**Обґрунтування:** Користувач вимагає перевірки реального UI через WebDriver, а не тільки функціонального покриття через Vitest-моки. Готовий ускладнити Rust-код заради тестованості (винести URL у конфіг). CI не потрібен — тільки ручний локальний запуск.

**Розглянуті альтернативи:**

- **Варіант A** — Vitest + Rust integration tests без Tauri runtime (відхилено: не відповідає вимозі реального UI).
- **Варіант C** — Playwright проти `bun run dev` з мок `window.__TAURI_INTERNALS__` (відхилено: не є справжнім e2e).

**Зачіпає:** `app/src-tauri/src/` (винесення Google URL у конфіг), `app/src-tauri/Cargo.toml` (залежності для тестів), майбутній `wdio.conf.js` та `tests/e2e/` у корені `app/`, системна залежність `webkit2gtk-driver` на Linux.
