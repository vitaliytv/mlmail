## ADR Стратегія витягування тіла email-повідомлення у Gmail MIME-парсері

## Context and Problem Statement

У застосунку MLMaiL деякі листи відображались із порожнім тілом. Причина: Rust-функція `extract_plain_text` у `app/src-tauri/src/gmail/message.rs` не охоплює всіх типових структур MIME-повідомлень, зокрема листів із порожнім `text/plain`-плейсхолдером, листів без текстових вузлів, і частин з не-UTF-8 кодуванням.

## Considered Options

- Повернути `text/plain` якщо знайдено, незалежно від вмісту (поточна реалізація — implicit)
- Перевіряти порожнечу `text/plain` перед поверненням; якщо порожній — спробувати `text/html`
- Додати тимчасове логування дерева `mimeType` для підтвердження причини, потім виправити
- Одразу виправити всі три виявлені баги: (A) пропускати порожній plain → fallback на HTML, (B) видаляти `<style>`/`<script>` перед strip_html, (C) читати `charset` з `Content-Type` і конвертувати з windows-1251/iso-8859-1

## Decision Outcome

Chosen option: "Провести діагностику + одразу виправити всі три баги", because власник проєкту отримав варіанти (A) тимчасовий лог або (B) одразу три виправлення, і сесія завершилась на очікуванні його відповіді — рішення ще не прийнято.

> **Примітка:** Сесія обірвалась до кінцевого вибору між діагностикою та негайним виправленням. Зафіксовано три підтверджені root causes:
>
> 1. `find_part("text/plain")` повертає порожній рядок, блокуючи fallback на HTML (`message.rs:30-38`)
> 2. `strip_html` regex `<[^>]+>` залишає порожнечу для листів-картинок без текстових вузлів (`message.rs:66-71`)
> 3. `String::from_utf8(bytes).ok()?` мовчки скидає не-UTF-8 частини; `charset` з `Content-Type` не читається (`message.rs:61-64`)

### Consequences

- Good, because transcript фіксує очікувану користь: виправлення усіх трьох причин усуне пусте тіло для маркетингових листів із порожнім plain, листів-картинок і листів у не-UTF-8 кодуванні.
- Bad, because transcript не містить підтверджених негативних наслідків.
