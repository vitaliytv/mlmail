# Міграція тестового стеку з bun:test на Vitest

**Status:** Accepted
**Date:** 2026-06-15

## Context and Problem Statement

Проєкт використовував `bun test` з `@happy-dom/global-registrator`-preload для тестування Vue-компонентів. Після синхронізації `@nitra/cursor` до v11 (для отримання правила `n-tool-surface.mdc`) Stop-хук почав регенеровувати `app/vitest.config.mjs` та `scripts/vitest.config.mjs`, несумісні з bun-test сетапом. Це ламало `knip`, `jscpd` і спричиняло постійний churn у робочому дереві.

## Considered Options

- Мігрувати на Vitest (узгодити з `@nitra/cursor` v11 і стеком `nitra/task`)
- Відкотити `@nitra/cursor` до `^1.27.5` і лишити `bun test`
- Розширити `knip`-ignore для vitest-файлів як паліатив, лишаючи `bun test`

## Decision Outcome

Chosen option: "Мігрувати на Vitest", because користувач явно сформулював «перемігровуємо на vitest»; це усуває корінь churn — v11-конфіги стають коректними й idempotent, Stop-хук більше не перегенеровує колізійні файли.

### Consequences

- Good, because `app/vitest.config.mjs` через `mergeConfig(vite.config.js callback, …)` дає нативну SFC-компіляцію та авто-імпорти без preload-хаків; `happy-dom.preload.js` і `?initial-state`-трюк більше не потрібні.
- Good, because `@stryker-mutator/vitest-runner` тепер коректно listed; `stryker.config.mjs` → `vitest-runner` + `perTest`.
- Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

Зміни: `app/vitest.config.mjs` (`mergeConfig` з callback-vite.config.js загорнутим у функцію, `environment: 'happy-dom'`); `app/stryker.config.mjs` → `testRunner: 'vitest'` + `perTest`; 5 тест-файлів — `bun:test` → `vitest` (`vi.hoisted`+`vi.mock`, `vi.resetModules()` замість `?initial-state`); видалено `app/test/happy-dom.preload.js`, `@happy-dom/global-registrator`, `@types/bun`, `@vue/compiler-sfc`; root devDeps += `vitest`, `@vitest/coverage-v8`, `@stryker-mutator/vitest-runner`; `scripts/package.json` += `vitest`. 85/85 тестів проходять.
