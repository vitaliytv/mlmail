## ADR: Архітектура Google OAuth у MLMaiL

**Контекст:** MLMaiL — крос-платформений Tauri 2 + Vue 3 застосунок (macOS і Android), якому потрібна Google OAuth-авторизація для Gmail API. Стандартна браузерна OAuth redirect не підходить через природу нативного застосунку.

**Рішення/Процедура/Факт:**
Реалізовано три пов'язані архітектурні рішення:

1. **Платформний OAuth flow** — macOS: Authorization Code + PKCE через TCP loopback (порт 0) + `tauri-plugin-opener` відкриває браузер; Android: Credential Manager + Authorization API через Tauri 2 mobile plugin (Kotlin `MlmailAuthPlugin`, `CredentialManagerFlow`, `AuthorizationFlow`).
2. **Token storage** — Keychain через `keyring` crate (macOS) та EncryptedSharedPreferences AES256_GCM (Android); Stronghold відкинуто як зайвий рівень складності.
3. **Rust як єдиний holder токенів** — Frontend (Vue) не бачить refresh token; Tauri-команди `auth_start_login`, `auth_get_access_token`, `auth_is_authenticated`, `auth_current_email`, `auth_logout`.

**Обґрунтування:** Loopback + PKCE — стандарт RFC 8252 для нативних desktop-apps (Chrome теж так). Credential Manager — рекомендований Google API з Android 14+. Rust-encapsulation токенів захищає від витоку через WebView JS-контекст. `keyring` crate — battle-tested, абстрагує Keychain API без сирих FFI. Stronghold (Tauri офіційний) — зайвий рівень без переваг над platform storage для single-tenant app.

**Розглянуті альтернативи:** AppAuth-Android (відкинуто — Java SDK, важча інтеграція з Rust); GoogleSignIn deprecated SDK; Stronghold (відкинуто); token exposure у JS (variant B/C — відкинуто).

**Зачіпає:** `app/src-tauri/src/auth/` (pkce.rs, state.rs, token_exchange.rs, id_token.rs, config.rs, error.rs, flow/macos.rs, flow/android.rs, storage/macos.rs, storage/android.rs, storage/in_memory.rs), `app/src/views/Login.vue`, `app/src/services/auth-store.js`, `app/src/i18n/auth-errors.js`, `docs/adr/ADR-0006-google-oauth.md`, C4-документація (02-containers.md, 03-components.md, 04-code.md, decisions.md).

## ADR: Читання Google OAuth Client IDs з `.env` у runtime

**Контекст:** Після заповнення `.env` застосунок повертав `401 invalid_client` — у бінарник було вшито плейсхолдер `REPLACE_ME_DESKTOP_CLIENT_ID`, бо `option_env!()` є compile-time макросом і cargo не бачив env vars під час збірки через `bun run tauri dev`.

**Рішення/Процедура/Факт:** Замінено `option_env!()` в `config.rs` на runtime читання через `std::env::var()` + `OnceLock<String>` кеш; `dotenvy` crate додано до `Cargo.toml` і викликається першим у `lib.rs::run()` через `dotenvy::dotenv().ok()`. Тест перевіряє, що відсутній env var повертає `Err(AuthError::Platform(...))`, а не panic.

**Обґрунтування:** `option_env!()` вимагає env var у середовищі CI/shell в момент `cargo build` — незручно для розробки (кожна зміна `.env` → `cargo clean`). Runtime читання дозволяє редагувати `.env` без перекомпіляції; `OnceLock` кешує після першого виклику для production-perf. `dotenvy` — стандартний Rust crate для `.env` файлів.

**Розглянуті альтернативи:** `option_env!()` compile-time (відкинуто — потребує `cargo clean` після кожної зміни), передавати env через shell перед запуском (відкинуто — fragile, незручно для розробників).

**Зачіпає:** `app/src-tauri/src/auth/config.rs`, `app/src-tauri/src/lib.rs`, `app/src-tauri/Cargo.toml`, `app/src-tauri/Cargo.lock`.
