# Стратегія витягування тіла email-повідомлення у Gmail MIME-парсері

**Status:** Accepted
**Date:** 2026-05-22

## Context and Problem Statement

У застосунку MLMaiL деякі листи відображались із порожнім тілом. Виявлено три root causes у `app/src-tauri/src/gmail/message.rs`:

1. `find_part("text/plain")` повертає порожній рядок, блокуючи HTML-фолбек (`message.rs:30-38`)
2. `strip_html` regex `<[^>]+>` залишає порожнечу для листів-картинок без текстових вузлів (`message.rs:66-71`)
3. `String::from_utf8(bytes).ok()?` мовчки скидає не-UTF-8 частини; `charset` з `Content-Type` не читається (`message.rs:61-64`)

## Considered Options

- Додати тимчасове логування дерева `mimeType` для підтвердження причини, потім виправити
- Одразу виправити всі три баги без попередньої діагностики

## Decision Outcome

Chosen option: "Провести діагностику через тимчасовий `eprintln!`-дамп, потім виправити всі три баги", because перед виправленням потрібно зібрати докази для конкретних листів щоб підтвердити гіпотези. Примітка: сесія завершилась до отримання остаточного вибору від користувача — рішення зафіксовано за контекстом наступної сесії.

### Consequences

- Good, because виправлення трьох root causes усуне порожнє тіло для маркетингових листів із порожнім `text/plain`-блоком, листів-картинок і листів у не-UTF-8 кодуванні.
- Bad, because transcript не містить підтверджених негативних наслідків.

## Update 2026-05-22

Діагностичний підхід (сесія `53ef6147`, 20:19): перед виправленням трьох root causes реалізовано тимчасовий дамп MIME-дерева.

Вибір `eprintln!` замість `log::debug!` обґрунтований тим, що у `app/src-tauri/src/lib.rs` не ініціалізовано жодного backend'у для crate `log` — макроси `log::*` були б беззвучні; `eprintln!` гарантовано виводиться у термінал `tauri dev`.

Змінені файли (тимчасово, позначені `// TEMP DEBUG (empty-body diagnosis) — remove after root cause confirmed`):

- `app/src-tauri/src/gmail/message.rs`: нова функція `debug_describe_payload()` — рекурсивно обходить MIME-дерево, виводить `mimeType`, розмір, наявність `body.data`, результат base64/UTF-8-декодування, 120-символьний preview, `Content-Type` з charset
- `app/src-tauri/src/gmail/mod.rs`: тригер у `get_message_at` — якщо `body_text.trim().is_empty()`, викликає `debug_describe_payload(payload)` через `eprintln!("[mlmail][TEMP-DEBUG] ...")`

Верифікація: `cargo check --lib` і `cargo clippy --lib` без нових помилок; єдине попередження clippy — у pre-existing коді в `auth`.
