# Стратегія витягування тіла email-повідомлення у Gmail MIME-парсері

**Status:** Accepted
**Date:** 2026-05-22

## Context and Problem Statement

У застосунку MLMaiL деякі листи відображались із порожнім тілом. Виявлено три root causes у `app/src-tauri/src/gmail/message.rs`:

1. `find_part("text/plain")` повертає порожній рядок, блокуючи HTML-фолбек (`message.rs:30-38`)
2. `strip_html` regex `<[^>]+>` залишає порожнечу для листів-картинок без текстових вузлів (`message.rs:66-71`)
3. `String::from_utf8(bytes).ok()?` мовчки скидає не-UTF-8 частини; `charset` з `Content-Type` не читається (`message.rs:61-64`)

## Considered Options

- Додати тимчасове логування дерева `mimeType` для підтвердження причини, потім виправити
- Одразу виправити всі три баги без попередньої діагностики

## Decision Outcome

Chosen option: "Провести діагностику через тимчасовий `eprintln!`-дамп, потім виправити всі три баги", because перед виправленням потрібно зібрати докази для конкретних листів щоб підтвердити гіпотези. Примітка: сесія завершилась до отримання остаточного вибору від користувача — рішення зафіксовано за контекстом наступної сесії.

### Consequences

- Good, because виправлення трьох root causes усуне порожнє тіло для маркетингових листів із порожнім `text/plain`-блоком, листів-картинок і листів у не-UTF-8 кодуванні.
- Bad, because transcript не містить підтверджених негативних наслідків.

## Update 2026-05-22

### Виправлення base64url-декодування: `DecodePaddingMode::Indifferent`

`body.data` у відповідях Gmail API містить стандартний base64url із `=`-паддингом. Попередній двигун `URL_SAFE_NO_PAD` відхиляв паддинг — тіло листів повертало порожній рядок (підтверджено `eprintln!`-дампом: `b64=ERR(Invalid padding)`). Кастомний `base64::engine::GeneralPurpose` з `DecodePaddingMode::Indifferent` приймає обидва формати. Залежність: `base64 = "0.22"` (вже існувала). Файл: `app/src-tauri/src/gmail/message.rs` — константа `BASE64URL`, функція `decode_part()`. Тест: `gmail::message::tests::extract_plain_text_decodes_padded_base64url` перейшов з RED на GREEN.

### Підтримка нон-UTF-8 charset через `encoding_rs`

Gmail API повертає MIME-частини з різними кодуваннями (`ISO-8859-1`, `windows-1251`, `gb2312`, `shift_jis` тощо). Попередня `decode_base64url` викликала `String::from_utf8(bytes).ok()?`, яка повертала `None` для нон-UTF-8 байтів. Обрано `encoding_rs = "0.8"` (Mozilla), бо він охоплює всі Web-кодування; `charset` параметр із `Content-Type` тепер читається функцією `part_charset()`; fallback — UTF-8 lossy. Файли: `app/src-tauri/Cargo.toml`, `app/src-tauri/src/gmail/message.rs`. Тести `extract_plain_text_decodes_iso_8859_1_body` та `..._windows_1251_body` перейшли з RED на GREEN; загалом 93/93 тестів проходять. Версія `app` `0.1.2` → `0.1.3`.
