## Проблеми

### 1. `store.status = 'loading'` — пряма мутація, яка не працює в Pinia composition store

`useAuthStore` — це composition store (`defineStore('auth', () => {...})`). У таких сторах стан — це `ref`, і пряме присвоєння `store.status = 'loading'` **не змінює** реактивний `ref.value`. Треба або `store.status = ...` через `$patch`, або замокати сам стор.

Правильно:

```js
store.$patch({ status: 'loading' })
// або
vi.spyOn(store, 'status', 'get').mockReturnValue('loading')
```

**Але фактично** — Pinia `$patch` для composition stores теж не завжди спрацьовує зі звичайними `ref`. Найнадійніший варіант — мокати весь стор:

```js
vi.mock('../services/auth-store.js', () => ({
  useAuthStore: () => ({ status: 'loading', error: null, email: null })
}))
```

### 2. `data-testid` атрибутів нема в `Login.vue`

Тест шукає `[data-testid="login-btn"]` та `[data-testid="logout-btn"]`, але в `Login.vue` їх немає. Тести впадуть.

## [Knowledge] `vi.hoisted()` обов'язковий для mock-об'єктів з `vi.fn()` поза фабрикою

**Контекст:** У Vitest `vi.mock()` hoistується компілятором на початок файлу, виконуючись до решти коду. Якщо mock-об'єкт з `vi.fn()` оголошений як звичайна `const` поза фабрикою `vi.mock(...)`, виникає помилка `Cannot use vi.fn() outside of a mock factory`.

**Рішення/Процедура/Факт:** Оголошувати shared mock-об'єкти через `vi.hoisted(() => ({...}))` — це спеціальний API Vitest, який виконує ініціалізатор у тій самій фазі hoisting що й `vi.mock`.

**Обґрунтування:** `vi.hoisted` гарантує що `vi.fn()` викликається вже після ініціалізації Vitest runtime, а змінна доступна у фабриці `vi.mock`.

**Розглянуті альтернативи:** Перемістити всі `vi.fn()` всередину фабрики `vi.mock` — але тоді немає доступу до них у тестах для перевірки викликів.

**Зачіпає:** `app/src/views/Login.test.js`, всі майбутні Vitest-тести з shared mock-об'єктами.
