## ADR Автоматичний реліз Tauri (DMG + APK) через git-тег

## Context and Problem Statement
Monorepo mlmail (Tauri 2, bun-workspaces) потребував автоматизованого релізного пайплайну для публікації macOS DMG і Android APK з єдиного git-тегу без ручного втручання.

## Considered Options
* Тригер на push гілки `main`
* Тригер на push тегу `v*` з окремими джобами `build-desktop` і `build-android`

## Decision Outcome
Chosen option: "Тригер на push тегу `v*` з окремими джобами", because тег є єдиним джерелом правди для версії релізу; джоба `create-release` утворює GitHub Release, потім `build-desktop` (macOS universal DMG через `tauri-apps/tauri-action@v0`) і `build-android` (APK через `bun --cwd=app run tauri android build --apk`) запускаються паралельно й заливають артефакти через `gh release upload`.

### Consequences
* Good, because transcript фіксує очікувану користь: перший повний прогін v0.1.3 завершився за 21m32s; реліз містить `mlmail_0.1.3_universal.dmg` і `app-universal-release.apk`.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Файли: `.github/workflows/release.yml` (mlmail, myshare, task). Крок синку версії: `VER="${GITHUB_REF_NAME#v}"; jq --arg v "$VER" '.version=$v' tauri.conf.json > tmp && mv tmp tauri.conf.json` виконується в обох джобах перед збіркою. ## ADR Виключення `Swatinem/rust-cache` і `softprops/action-gh-release` з release-workflow

## Context and Problem Statement
Перший варіант `release.yml` містив `Swatinem/rust-cache@v2` і `softprops/action-gh-release@v2`; `zizmor` підняв security-findings щодо них у release-контексті.

## Considered Options
* Залишити rust-cache і сторонній upload-екшен
* Прибрати rust-cache; завантажувати артефакти через вбудований `gh release upload`

## Decision Outcome
Chosen option: "Прибрати rust-cache; завантажувати через `gh release upload`", because `zizmor` класифікував `rust-cache` у release-workflow як cache-poisoning ризик; `gh` вже присутній на runner і не вносить нову залежність.

### Consequences
* Good, because transcript фіксує очікувану користь: `zizmor` перестав повідомляти findings після видалення (`No findings to report. Good job! (31 suppressed)`).
* Bad, because час збірки Rust не кешується — кожен реліз компілюється з нуля (~20 хв). Transcript не містить підтверджених негативних наслідків щодо security-аспекту.

## More Information
Правило `zizmor.yml`: `unpinned-uses: ref-pin`. Після зміни лінт-команда `bun run lint-ga` (actionlint + zizmor) проходить чисто для `release.yml` і `clean-merged-branch.yml`.

## ADR PKCS12 keystore: `keyPassword = storePassword` у CI

## Context and Problem Statement
Android release-підпис у CI стабільно падав з `KeytoolException: Failed to read key … Get Key failed: Given final block not properly padded`, навіть після оновлення `ANDROID_KEY_PASSWORD` у Infisical.

## Considered Options
* Передавати `ANDROID_KEY_PASSWORD` як окремий секрет (відмінний від storePassword)
* Брати `keyPassword` зі `storePassword` безпосередньо у `keystore.properties`

## Decision Outcome
Chosen option: "Брати `keyPassword` зі `storePassword` безпосередньо у `keystore.properties`", because `openssl pkcs12` показав, що приватний ключ розшифровується *тільки* паролем store (`keys: 1`), а будь-яким іншим — ні (`keys: 0`); сучасний `keytool` під час генерації PKCS12 ігнорує окремий `-keypass` і шифрує ключ тим самим паролем що й сховище.

### Consequences
* Good, because transcript фіксує очікувану користь: після заміни `keyPassword=$ANDROID_KEY_PASSWORD` на `keyPassword=$ANDROID_KEYSTORE_PASSWORD` у кроці «Configure signing», `build-android` v0.1.3 пройшов повністю (`Build signed APK ✅`, `Upload APK ✅`).
* Bad, because якщо в майбутньому keystore буде перегенерований у форматі JKS (не PKCS12), де `-keypass` зберігається окремо, цей патерн зламається без попередження.

## More Information
Перевірка: `openssl pkcs12 -in mlmail-release.jks -passin pass:<SP> -nokeys -noout` → `certs: 1`; `openssl pkcs12 -in … -nodes -noout -passin pass:<SP>` → `keys: 1`; той самий виклик з `-passin pass:<KP>` → `keys: 0`. Виправлення в `release.yml` (рядок кроку «Configure signing»): `echo "keyPassword=$ANDROID_KEYSTORE_PASSWORD"`. Стосується і mlmail, і myshare.

## ADR `bun --cwd=app` (зі знаком `=`) у CI-командах

## Context and Problem Statement
У CI-джобі `build-android` команда `bun --cwd app run tauri android build --apk` завершувалась за ~1m26s без жодної Rust-компіляції: `Build signed APK` позначалось `✓`, але APK не зʼявлявся.

## Considered Options
* `bun --cwd app run …` (пробіл між прапором і значенням)
* `bun --cwd=app run …` (знак `=`)

## Decision Outcome
Chosen option: "`bun --cwd=app run …`", because локальна перевірка підтвердила: `bun --cwd app` інтерпретує `app` як позиційний аргумент (файл), друкує власний usage і виходить; `bun --cwd=app run tauri --version` повертає `tauri-cli 2.11.2`.

### Consequences
* Good, because transcript фіксує очікувану користь: після виправлення CI реально запустив Rust-компіляцію (~20 хв) і отримав APK.
* Bad, because transcript не містить підтверджених негативних наслідків.

## More Information
Виправлено у `.github/workflows/release.yml` mlmail (рядок з `run: bun --cwd=app run tauri android build --apk`) і у відповідному файлі myshare. У `nitra/task` виправлено лише коментар (Android ще не ініціалізований).
