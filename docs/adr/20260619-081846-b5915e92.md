Так, Infisical підходить — і навіть дуже добре для IT-команди з CI/CD. Ось що конкретно можна зробити:

## Що зберігати в Infisical

Keystore — бінарний файл, тому зберігай його як base64-рядок:

```bash
# конвертувати .jks у base64 і скопіювати
base64 -i mlmail-release.jks | pbcopy
```

Потім у Infisical-проєкті (напр. `mlmail/production`) — 4 секрети:

```
ANDROID_KEYSTORE_BASE64    # base64 вміст .jks файлу
ANDROID_KEYSTORE_PASSWORD
ANDROID_KEY_ALIAS          # mlmail
ANDROID_KEY_PASSWORD
```

## Інтеграція з GitHub Actions

Infisical має офіційний `infisical/secrets-action`:

```yaml
# у release.yml — замість ручного gh secret set
- uses: Infisical/secrets-sync-action@v1
with:
client-id: ${{ secrets.INFISICAL_CLIENT_ID }}
client-secret: ${{ secrets.INFISICAL_CLIENT_SECRET }}
project-slug: mlmail
env-slug: production
secret-path: /
```

Або через CLI прямо в CI-кроці:

```bash
infisical run --projectId=<id> --env=production -- \
./gradlew bundleRelease
```

## Мінус

Infisical не зберігає бінарні файли нативно — `.jks` доведеться тримати як base64-рядок. 1Password зберігає сам файл як вкладення. Для автоматизації це не різниця, але для ручного відновлення трохи незручніше.

**Висновок:** Якщо команда вже розглядає Infisical або хоче self-hosted рішення — чудовий вибір. Якщо потрібно ще й ручне використання (розробник відновлює keystore на новій машині) — 1Password трохи зручніший через нативні файли.
