## ADR Відображення кількості листів INBOX на стартовому екрані

**Контекст:** MLMaiL отримала потребу показувати на стартовому екрані (Login.vue) точну кількість листів у скриньці одразу після успішного входу, використовуючи вже наявний scope `gmail.modify`.

**Рішення/Процедура/Факт:** Для отримання числа обрано ендпоінт `GET /gmail/v1/users/me/labels/INBOX` (`users.labels.get?id=INBOX`), який повертає поле `messagesTotal` (усі листи INBOX). HTTP-виклик виконується в новому Rust-модулі `app/src-tauri/src/gmail/mod.rs` через команду `gmail_inbox_count(state) -> Result<u64, GmailError>`. Access token бере з `AuthState` через існуючу функцію `auth_get_access_token`; на 401 повертає `GmailError::ReauthRequired`, на мережеву помилку — `GmailError::Network`. Vue-частина: `auth-store.js` отримує новий `ref inboxCount` та `refreshInboxCount()`, які викликаються автоматично після `initialize()`/`login()`. Login.vue рендерить число під рядком з email; під час завантаження — плейсхолдер «…»; помилка — і18n-рядок. Оновлено документацію `docs/ci4/03-components.md` і `docs/ci4/04-code.md`. Спека закомічена у `docs/superpowers/specs/2026-05-14-inbox-count-design.md`.

**Обґрунтування:** `users.labels.get?id=INBOX` — канонічний однозапитний шлях з точним значенням; `users.getProfile` повертає загальний `messagesTotal` по всій скриньці (не тільки INBOX), а `messages.list resultSizeEstimate` — приблизне. Розміщення HTTP-виклику в Rust узгоджується з існуючим патерном команд `auth_*`, не виносить access token у JS-рівень і виключає CORS-проблеми у WebView.

**Розглянуті альтернативи:**

- `users.getProfile` — повертає кількість по всій скриньці, не тільки INBOX
- `messages.list?labelIds=INBOX&maxResults=1` з `resultSizeEstimate` — приблизне, неточне
- Прямий `fetch` з Vue — потребує передачі токена у JS, порушує наявний патерн
