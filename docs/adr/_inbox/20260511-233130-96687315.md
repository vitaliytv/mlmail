## ADR Архітектура Google OAuth у MLMaiL

**Контекст:** Tauri 2 + Vue 3 застосунок MLMaiL потребує OAuth 2.0 авторизації через Google для доступу до Gmail API на двох платформах — macOS і Android. Необхідно обрати flow, secure storage для токенів і межу між фронтендом і бекендом.

**Рішення/Процедура/Факт:**
Реалізовано три незалежних рішення:

1. **OAuth flow по платформах:** macOS — loopback HTTP server (tokio + hyper, порт 0) + Authorization Code + PKCE + `tauri-plugin-opener` для браузера; Android — Credential Manager + Google Authorization API через Tauri 2 mobile plugin → Kotlin (`CredentialManagerFlow.kt`, `AuthorizationFlow.kt`), server auth code обмінюється на токени на стороні Rust.

2. **Token storage:** macOS — System Keychain через `keyring` crate (`apple-native` feature); Android — `EncryptedSharedPreferences` (AES256-GCM, master key у Android Keystore). Таuri Stronghold відкинуто як зайвий шар. Refresh token зберігається лише в Rust/platform storage — JS-рівень ніколи не отримує токен сирцем.

3. **Межа Frontend/Backend:** Vue отримує `{ email, isAuthenticated }` і викликає 5 Tauri-команд (`auth_start_login`, `auth_get_access_token`, `auth_is_authenticated`, `auth_current_email`, `auth_logout`). Access token залишається в `Mutex<AuthState>` Rust-процесу і не серіалізується у JS-контекст.

Нові файли: `app/src-tauri/src/auth/` (13 модулів), `app/src/views/Login.vue`, `app/src/services/auth-store.js`, `app/src/i18n/auth-errors.js`, 4 Kotlin-файли в `gen/android/.../auth/`. Документацію оновлено: `docs/ci4/02-containers.md`, `03-components.md`, `04-code.md`, `decisions.md`, новий `docs/adr/ADR-0006-google-oauth.md`.

**Обґрунтування:**
- Loopback PKCE на macOS: Desktop-type OAuth client Google приймає `http://127.0.0.1` будь-який порт без реєстрації redirect URI; не потребує кастомних URI-схем; ~80 рядків Rust замість community plugin.
- Credential Manager + Authorization API на Android: це поточний рекомендований Google підхід (замість deprecated `GoogleSignIn`); `forceCodeForRefreshToken=true` гарантує видачу server auth code.
- Rust як єдиний holder токенів: усуває ризик витоку через WebView, дозволяє безпечно проксіювати Gmail API виклики (наступна ітерація).
- `keyring` crate (apple-native): підтримує Security framework напряму без D-Bus залежностей; cross-platform trait дозволяє підставити `InMemoryStorage` у тестах.

**Розглянуті альтернативи:**
- AppAuth-Android: окрема бібліотека, зайва складність, застарілий підхід.
- Tauri Stronghold: додатковий рівень шифрування поверх platform Keystore — надлишковий.
- Зберігати access token у Vue ref: Frontend тримав би секрет у WebView-пам'яті — відкинуто.
- `security-framework` crate напряму замість `keyring`: більше boilerplate без переваг у функціональності.
