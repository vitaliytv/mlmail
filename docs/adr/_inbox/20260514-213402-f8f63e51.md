## Knowledge Tauri — відсутність web-таргета та архітектурні наслідки

**Контекст:** Команда використовує Tauri для desktop+mobile у проєкті MLMaiL і потребувала пояснення, чи можна ту саму Tauri-збірку задеплоїти у браузер, як це дозволяє Capacitor.

**Рішення/Процедура/Факт:** Tauri не має `web`-платформи. Capacitor має `@capacitor/web` з no-op заглушками — одна збірка працює і у WebView, і у браузері. Для Tauri frontend (Vue/React) деплоїться окремо через Vite як PWA. `#[tauri::command]` у браузері не існує; потрібен тонкий шар `platform.ts`: на Tauri — `invoke('cmd', args)`, на web — `fetch('/api/cmd')` або WASM-виклик. Rust-бізнес-логіка для web компілюється у WASM через `wasm-bindgen` — це покриває парсери, crypto, ембединги, але не FS, keychain і OS API (там потрібен бекенд). `candle` і `ort` мають WASM-збірки → inference у браузері без GPU, для важких моделей — все одно сервер. На Android Rust компілюється до NDK через JNI напряму, без Java/Kotlin у бізнес-логіці.

**Обґрунтування:** Tauri позиціонується як shell для desktop+mobile, а не як cross-platform web framework. Це дозволяє мінімальний бінарник (3–10 MB проти 100+ MB у Electron/Capacitor desktop), нативний доступ до keychain і OS API без JS-моста, і можливість запускати локальні AI-моделі (llama.cpp, candle, ort) у Rust-процесі без IPC bottleneck.

**Розглянуті альтернативи:** Capacitor — виграє коли команда суто JS, потрібен Cordova ecosystem або проєкт suто mobile-first без важких обчислень.

**Зачіпає:** `app/src/platform.ts` (шар абстракції для IPC), `src-tauri/src/` (Rust-команди), потенційний `wasm/` пакет для web-таргета; архітектурні рішення щодо деплою MLMaiL у браузер.
