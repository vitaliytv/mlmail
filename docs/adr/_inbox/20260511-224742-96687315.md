## ADR Google OAuth архітектура MLMaiL (macOS + Android)

**Контекст:** MLMaiL потребує доступу до Gmail API з scope `gmail.modify` на двох платформах — macOS (Tauri desktop) і Android (Tauri mobile). Потрібно вибрати OAuth mechanism, де зберігати токени і як розподілити відповідальність між Rust-бекендом і Vue-фронтендом.

**Рішення/Процедура/Факт:**
- **OAuth mechanism**: macOS — Authorization Code + PKCE через loopback `TcpListener` на `127.0.0.1:RANDOM_PORT` + `tauri-plugin-opener`; Android — Credential Manager + Authorization API (через JNI-міст до Kotlin `MlmailAuth.signInAndAuthorize`). Android не використовує PKCE — Google підписує запит SHA-1 fingerprint APK, `redirect_uri=""`.
- **Token storage**: macOS — `keyring` crate з `apple-native` feature (Keychain Services); Android — `EncryptedSharedPreferences` (AES256_GCM master key у Android Keystore) через JNI. Ключ сховища: `com.vitaliytv.mlmail` / `google.refresh_token`.
- **Token surface**: Rust тримає `refresh_token` (у сховищі) і `access_token` (in-memory у `AuthState`). Vue отримує access token лише через Tauri-команду `auth_get_access_token`. Автоматичний refresh при `expires_at < now + 30s`; `invalid_grant` → `AuthError::ReauthRequired` → фронтенд повертається на Login.
- **Client IDs**: compile-time константи через `option_env!` у `auth/config.rs`; локально — `app/src-tauri/.env` (gitignore'd).
- Spec: `docs/superpowers/specs/2026-05-11-google-oauth-design.md`; Plan: `docs/superpowers/plans/2026-05-11-google-oauth-plan.md` (12 tasks).

**Обґрунтування:** Loopback на macOS — Google-рекомендований підхід для Desktop OAuth clients без custom URI scheme. Credential Manager + Authorization API на Android — поточний (2025) Google-рекомендований підхід замість deprecated `GoogleSignIn`; AppAuth-Android не обраний бо Credential Manager дає нативний bottom sheet UX. Rust як єдиний holder токенів — токени не торкаються JS-пам'яті, JS sandbox не може їх витягти. `keyring`/`EncryptedSharedPreferences` замість Tauri Stronghold — менше коду, нативна інтеграція з OS secure enclave без додаткового шару шифрування.

**Розглянуті альтернативи:** AppAuth-Android (відкинуто — Credential Manager новіший і офіційний); deprecated GoogleSignIn (відкинуто); Tauri Stronghold (відкинуто — зайвий рівень поверх OS keystore); platform-specific deep links замість loopback (складніше, потребує реєстрації custom scheme); token surface варіанти де Vue тримає токени (відкинуто — security).
