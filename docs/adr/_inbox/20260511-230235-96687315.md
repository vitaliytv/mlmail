# Writing Implementation Plans

Create a detailed, actionable implementation plan from a spec.

## When to Invoke

This skill is invoked:

- After brainstorming produces a validated spec
- When a user says "write a plan", "create a plan", "make a plan", or similar
- Before beginning any multi-step implementation task

## Skill Process

1. Re-read the spec document carefully
2. Identify all work to be done, any unknowns, and implicit dependencies
3. Check for existing implementations to avoid duplication
4. Map out task dependencies to enable maximum parallelism
5. Write a detailed plan document
6. Present a concise summary to the user

## Plan Format

Save to `docs/superpowers/plans/YYYY-MM-DD-<topic>.md`:

```markdown
# Implementation Plan: [Topic]

Spec: [link to spec]

## Overview

[Two to three sentence summary of what this plan implements and why]

## Tasks

[Tasks listed with dependencies noted, roughly ordered from most foundational to highest level]

### Task N: [short verb phrase]

**What:** [what this task does and why it's needed]
**How:** [specific implementation steps]
**Depends on:** [task numbers, or "nothing"]
**Must complete before:** [task numbers, or "nothing"]
**Done when:** [specific, testable criteria]
```

## Clarifications

**Completeness over conciseness** - Include implementation details, not just summaries. Each step should be unambiguous.

**Avoid the word "existing"** - Instead of "update existing X", write "update X in [file] at [location]". Describe what changes, not what was already there.

**Phasing** - If the plan involves many tasks, clearly indicate which tasks should be tackled first (the critical path).

**Parallelism** - Where multiple tasks have no dependencies between them, note them explicitly as parallelizable.

**Plan document stays in docs/superpowers/plans/**, commited via git.

## ADR Архітектура Google OAuth авторизації MLMaiL

**Контекст:** MLMaiL потребує Google OAuth авторизації для доступу до Gmail API одночасно на macOS Desktop (Tauri) і Android, з безпечним зберіганням refresh token між сесіями. У C4-документації Auth Component і Auth Store були позначені як `planned` — потрібно було обрати конкретний flow і стек.

**Рішення/Процедура/Факт:** Обрано Authorization Code + PKCE через loopback redirect (127.0.0.1:random port) на macOS з `tauri-plugin-opener` для запуску браузера; на Android — Google Credential Manager + AuthorizationClient (play-services-auth) через JNI-міст з Rust до Kotlin. Refresh token зберігається у Keychain (macOS) через `keyring` crate і EncryptedSharedPreferences (Android) через JNI-виклики до Kotlin `SecureStore`. Rust є єдиним holder-ом токенів — Vue отримує лише `AuthSession { email }` і може запитати access token через `auth_get_access_token` Tauri command. Vue Login-екран локалізований українською через `vue-i18n@9`. Spec: `docs/superpowers/specs/2026-05-11-google-oauth-design.md`. План: `docs/superpowers/plans/2026-05-11-google-oauth.md`.

**Обґрунтування:** Loopback на macOS — рекомендований Google підхід для Desktop installed apps (не потребує кастомного URI scheme і не вимагає client_secret у коді). Credential Manager + AuthorizationClient на Android — актуальний Google SDK замість deprecated GoogleSignIn; дає server auth code для сервер-сайд token exchange, що уніфікує refresh token flow між платформами. Keychain + EncryptedSharedPreferences — platform-native сховища з апаратним захистом, рекомендовані Google; `keyring` crate з `apple-native` feature уникає D-Bus залежностей. Token isolation у Rust — mінімізує поверхню атаки: JS-код не тримає довгоживучих токенів.

**Розглянуті альтернативи:** AppAuth-Android (додатковий dependency, схожа складність JNI, відкинуто); `tauri-plugin-oauth` community plugin (менший контроль, зайва абстракція, відкинуто); Tauri Stronghold для token storage (зайвий рівень шифрування поверх системного Keychain, відкинуто); token в JS пам'яті / localStorage (небезпечно, відкинуто); custom URI scheme замість loopback (потребує реєстрації схеми в Info.plist, складніша конфігурація Google Console, відкинуто).
