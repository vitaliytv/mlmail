## ADR Google OAuth авторизація для MLMaiL (macOS + Android)
**Контекст:** MLMaiL — Tauri 2 + Vue 3 застосунок, що потребує доступу до Gmail API. Потрібна OAuth-авторизація через Google на двох платформах (macOS і Android) з безпечним збереженням refresh_token.
**Рішення/Процедура/Факт:** macOS — loopback HTTP + PKCE + системний браузер через `tauri-plugin-opener`; refresh_token зберігається у Keychain через `keyring` crate. Android — Credential Manager + `AuthorizationFlow` через Activity result; refresh_token зберігається у `EncryptedSharedPreferences`. Rust реалізує 5 Tauri-команд: `auth_start_login`, `auth_get_access_token`, `auth_is_authenticated`, `auth_current_email`, `auth_logout`. Vue Login-екран — українська мова, `auth-store.js` singleton composable, `i18n/auth-errors.js` мапить Rust `kind` enum → рядок. Scope: лише auth flow + token storage (без Gmail client / inbox UI). Реалізовано через TDD (54 тести: 32 Rust + 22 JS).
**Обґрунтування:** Loopback замість `custom-scheme` — надійніший на macOS без додаткових entitlements. `keyring` crate — крос-платформна абстракція над Keychain. Inline i18n без `vue-i18n` — достатньо для одного модуля, без зайвої залежності. Rust повертає лише `kind` (англійський enum), Vue відповідає за локалізацію — розділення відповідальностей.
**Розглянуті альтернативи:** Custom URI scheme (відхилено — складніше налаштування); `security-framework` напряму замість `keyring` (відхилено — менш portable); сирий JNI замість Tauri 2 mobile plugin (відхилено — офіційно підтримуваний шлях чистіший); `vue-i18n` для локалізації (відхилено — надмірно для одного auth-модуля).
**Зачіпає:** `app/src-tauri/src/auth/` (13 файлів: `pkce.rs`, `id_token.rs`, `state.rs`, `token_exchange.rs`, `config.rs`, `error.rs`, `storage/`, `flow/`), `app/src/views/Login.vue`, `app/src/services/auth-store.js`, `app/src/i18n/auth-errors.js`, `app/src-tauri/gen/android/.../auth/` (4 Kotlin файли), `Cargo.toml`, `app/package.json`.

## ADR Runtime читання OAuth Client IDs через `dotenvy`
**Контекст:** Після заповнення `.env` реальними Client ID застосунок повертав `401 invalid_client` — у бінарнику залишався рядок `REPLACE_ME_DESKTOP_CLIENT_ID`. Причина: `option_env!()` — compile-time макрос, який читає env на момент `cargo build`, а не при запуску.
**Рішення/Процедура/Факт:** `config.rs` переписано з `option_env!()` + константи → `std::env::var()` + `OnceLock<String>` (ліниво кешовані функції). `dotenvy::dotenv()` викликається у `lib.rs::run()` перед ініціалізацією Tauri. Змінені файли: `Cargo.toml` (`dotenvy = "0.15.7"`), `auth/config.rs`, `auth/mod.rs` (`DESKTOP_CLIENT_ID` → `desktop_client_id()`). Після зміни `.env` — просто перезапустити додаток без `cargo clean`.
**Обґрунтування:** Compile-time змінна робить `.env` практично марним (потрібен `cargo clean` + rebuild після кожної зміни). `dotenvy` + `OnceLock` дає runtime-читання без performance overhead при кожному виклику.
**Розглянуті альтернативи:** Compile-time `option_env!` (відхилено — вимагає rebuild після `.env`-змін); передача через `tauri::Context` / конфіг (відхилено — складніше, надмірно для трьох змінних).
**Зачіпає:** `app/src-tauri/src/auth/config.rs`, `app/src-tauri/src/auth/mod.rs`, `app/src-tauri/Cargo.toml`, `app/src-tauri/src/lib.rs`.

## Knowledge OAuth Client ID для native-app — не секрет, можна в приватний git
**Контекст:** При вирішенні, чи додавати `app/src-tauri/.env` з трьома Google OAuth Client IDs до gitignore або до репозиторію.
**Рішення/Процедура/Факт:** Google OAuth Client ID для native-додатку (`*.apps.googleusercontent.com`) є публічним ідентифікатором — він передається відкритим текстом у кожному OAuth-запиті в мережі і тривіально витягується зі збірки (`strings mlmail`). Google не вважає його секретом. `.env` зі ЛИШЕ Client IDs (без Client Secret) безпечно зберігати у приватному репо. `.env` знято з `.gitignore` і закомічено (commit `244001f`).
**Обґрунтування:** Репо `vitaliytv/mlmail` на GitHub — приватне (`isPrivate: true`). Не зберігати `.env` у git означає ускладнити onboarding без жодного реального виграшу в безпеці.
**Розглянуті альтернативи:** Тримати у `.gitignore` (відхилено — надлишковий захист; будь-який `strings binary` однаково показав би ці значення).
**Зачіпає:** `.gitignore`, `app/src-tauri/.env`, `app/src-tauri/.env.example`.

## Runbook Google OAuth consent screen через gcloud для Internal (Workspace org)
**Контекст:** Google Cloud Console не дозволяє створити OAuth consent screen типу "Internal" через UI без Workspace, але `gcloud iap oauth-brands create` підтримує це на рівні CLI — лише для проєктів у Workspace-організації.
**Рішення/Процедура/Факт:** 1. `gcloud organizations list` — знайти org ID. 2. `gcloud projects describe <project>` — перевірити, що parent є organization. 3. `gcloud iap oauth-brands create --application_title="MLMaiL" --support_email="..."` — створює brand з `orgInternalOnly: true` (лише `*@nitralabs.com` зможуть логінитися). 4. Три OAuth clientIDs (Desktop, Android, Web) — лише через UI: `console.cloud.google.com/apis/credentials`. API IAP OAuth Admin **deprecated з березня 2026** — для clientIDs gcloud більше не варіант.
**Обґрунтування:** Internal-тип знімає вимогу Google App Verification для sensitive scopes (зокрема `gmail.modify`), що критично на ранньому етапі до публічного релізу.
**Розглянуті альтернативи:** External тип з Test Users (відхилено — 100-user cap і потребує verification перед продакшном); продовжувати через UI (заняло б більше часу і потребувало б окремого login).
**Зачіпає:** Google Cloud project `vitaliytv`, три OAuth Client IDs у `app/src-tauri/.env`.
