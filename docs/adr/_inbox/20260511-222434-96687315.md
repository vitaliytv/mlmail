## ADR Архітектура Google OAuth 2.0 авторизації MLMaiL (macOS + Android)

**Контекст:** MLMaiL потребує доступу до Gmail API з scope `gmail.modify`. Застосунок розгортається на macOS і Android, що вимагає різних OAuth redirect механізмів і сховищ для refresh token.
**Рішення/Процедура/Факт:** Обрано platform-native підхід: macOS — loopback `TcpListener` на `127.0.0.1:0` з OAuth Desktop App client; Android — Google Credential Manager + Authorization API через Tauri mobile plugin (`AuthPlugin.kt`). Rust-шар тримає всі токени (access token in-memory в `AuthState`, refresh token у Keychain / `EncryptedSharedPreferences`). Vue Auth Store зберігає лише `isAuthenticated` і `userEmail`. Client IDs зберігаються у `tauri.conf.json` → секція `plugins.mlmail` (публічні значення, комітяться в репо). PKCE (SHA-256 code_challenge) обов'язковий для обох платформ.
**Обґрунтування:** Desktop App client + loopback = жодного реєстрованого redirect URI (Google wildcard port). Credential Manager + Authorization API = нативний UX на Android без кастомного Custom Tab. Rust-only токени = JS-код ніколи не бачить `access_token`/`refresh_token`, мінімальна attack surface. `plugins.mlmail` у `tauri.conf.json` — єдиний механізм Tauri 2 для довільних app-рівневих конфігурацій без env-файлів, читається через `app.config().plugins.0.get("mlmail")`.
**Розглянуті альтернативи:** loopback (macOS) + deep link (Android) — відкинуто на користь native SDK; AppAuth-Android — відкинуто на користь Credential Manager (Google-recommended 2024+); env-файли для Client ID — відкинуто (Client ID публічні, env-файли ускладнюють конфігурацію).
**Зачіпає:** `app/src-tauri/tauri.conf.json`, `app/src-tauri/src/auth/` (новий модуль), `app/src-tauri/gen/android/app/` (AuthPlugin.kt, build.gradle.kts), `app/src/` (vue-router, auth-store, Login.vue, App.vue).
