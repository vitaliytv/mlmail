## ADR Google OAuth 2.0 PKCE для MLMaiL (macOS + Android)

**Контекст:** MLMaiL — Tauri 2 + Vue 3 додаток (macOS + Android). Потрібна авторизація Google-акаунту щоб звертатись до Gmail API. Кожна платформа має власні обмеження: macOS не має нативного Credential Manager-а, Android — не може відкрити loopback-порт.

**Рішення/Процедура/Факт:**

- macOS: loopback HTTP сервер на випадковому порту приймає redirect від браузера; PKCE-пара `(verifier, challenge)` генерується у `auth/pkce.rs`; access/refresh токени зберігаються через `keyring` crate у macOS Keychain.
- Android: Kotlin `CredentialManagerFlow` через Google `GetSignInWithGoogleOption` отримує ID token; `AuthorizationFlow` через AppAuth отримує server auth code; зберігання — `EncryptedSharedPreferences` у `SecureStore.kt`; Rust викликає Kotlin через Tauri 2 mobile plugin (`MlmailAuthPlugin`).
- Спільне: `token_exchange.rs` обмінює code/refresh-token на `oauth2.googleapis.com/token`; `AuthState` зі структурованим полем `expiry: Option<Instant>` + 30-секундний буфер freshness; 5 Tauri-команд: `auth_start_login`, `auth_get_access_token`, `auth_is_authenticated`, `auth_current_email`, `auth_logout`.
- Vue-шар: `services/auth-store.js` (singleton composable, токени ніколи не потрапляють у JS), `views/Login.vue` (Ukrainian-only UI), `i18n/auth-errors.js` (Rust `kind` → Ukrainian рядок).

**Обґрунтування:** Loopback на macOS — єдиний flow без системного webview-компонента; `keyring` — стандарт Tauri-екосистеми для Keychain без прямого `security-framework` JNI. Credential Manager на Android — Google-рекомендований шлях замість `startActivityForResult` + custom scheme. Tauri mobile plugin pattern ізолює Kotlin від Rust без ручного JNI.

**Розглянуті альтернативи:**

- `security-framework` crate напряму (відхилено: `keyring` абстрагує і Keychain і Credential Manager через трейт)
- `raw JNI` з Rust (відхилено: Tauri 2 mobile plugin офіційно підтримуваний шлях)
- Зберігати access token у Vue-стані (відхилено: тільки Rust тримає токени в пам'яті)

**Зачіпає:** `app/src-tauri/src/auth/` (13 файлів), `gen/android/.../auth/` (4 Kotlin-файли), `app/src/services/auth-store.js`, `app/src/views/Login.vue`, `app/src/i18n/auth-errors.js`, `Cargo.toml`, `app/build.gradle.kts`

## ADR Runtime `.env` для Google OAuth Client IDs (dotenvy замість option_env!)

**Контекст:** Після першого запуску MLMaiL — `401 invalid_client` / "OAuth client was not found". Причина: `config.rs` використовував `option_env!("MLMAIL_GOOGLE_DESKTOP_CLIENT_ID")` — Rust compile-time макрос. Env var не була виставлена під час `cargo build`, тому в бінарнику назавжди опинився fallback `REPLACE_ME_DESKTOP_CLIENT_ID`.

**Рішення/Процедура/Факт:** Замінено `option_env!` → `std::env::var()` з кешуванням у `OnceLock<String>`. `dotenvy::dotenv()` викликається у `lib.rs` (точка входу Tauri) — зчитує `app/src-tauri/.env` при кожному старті. Три функції: `desktop_client_id()`, `android_client_id()`, `android_web_client_id()` замість констант. `.env` файл доданий до репо (private repo, Client IDs — публічні ідентифікатори native app, не секрети).

**Обґрунтування:** Native OAuth Client IDs не є секретами — вони видні у кожному HTTPS-запиті до `oauth2.googleapis.com` і витягуються `strings binary` будь-ким з доступом до бінарника. Зберігання у repо private-репо зручне і безпечне. Runtime-читання усуває проблему "збирав без env → вшитий placeholder".

**Розглянуті альтернативи:**

- `build.rs` + `cargo:rerun-if-env-changed` (не обговорювалось; ускладнює build-pipeline без переваг над dotenvy для цього випадку)
- Передавати env у shell перед `bun run tauri dev` (відхилено: крихко, потребує `cargo clean` після кожної зміни `.env`)

**Зачіпає:** `app/src-tauri/src/auth/config.rs`, `app/src-tauri/src/auth/mod.rs`, `app/src-tauri/src/lib.rs`, `Cargo.toml`, `Cargo.lock`, `app/src-tauri/.env`, `.gitignore`
