# Writing an Implementation Plan

Convert a spec or set of requirements into a clear, executable implementation plan, organized as ordered tasks with specific steps.

## Sub-skill requirement

Each task MUST name the sub-skill(s) that will execute it. Choose from:

- **superpowers:subagent-driven-development** — for tasks requiring the most powerful capability: creating new code, major structural changes, or complex multi-file edits
- **superpowers:executing-plans** — for simpler implementation tasks, with lighter overhead

Use **subagent-driven-development** when the task involves:

- Writing new files or significant new code
- Complex logic or algorithms
- Multi-step processes that span files or systems
- Anything where getting it right the first time matters

For everything else (small edits, config updates, documentation), use executing-plans.

## Process

1. **Read and understand** the spec or requirements document
2. **Define tasks** — ordered, parallel where possible, with specific steps and definitions of done
3. **Validate** — can each task be executed in isolation? Are dependencies explicit?
4. **Write** the plan to `docs/superpowers/plans/{date}-{topic}.md`
5. **Commit** the plan file
6. **Check it** — run a final self-review of the plan

## Task Format

Each task MUST have:

- A clear title and objective
- Specific steps (numbered)
- Definition of done (measurable)
- Sub-skill name(s)

If the plan has ordered tasks AND some parallelizable tasks, split it into "Phase" sections:

- **Phase 1:** Ordered sequential tasks (prerequisites for everything else)
- **Phase 2:** Parallel tasks that can run concurrently (same prerequisite set)
- **Phase 3:** Ordered sequential tasks that depend on Phase 2 completion

Put the sub-skill recommendation at the top of the plan for the agentic worker who will be implementing. They need to know which sub-skill they should use as they implement.

## Plan Principles

- **One concern per task** — each task should do one thing well
- **Specific file paths** — name the exact files to be created/modified
- **Testable outcomes** — definition of done should be verifiable
- **Right-sized tasks** — not too small (overhead), not too large (hard to review)
- **Dependencies explicit** — if task B requires task A, say so

## Self-review checklist

After writing the plan, run through each item:

- [ ] Every task has a clear definition of done
- [ ] No task has ambiguous steps ("set up X", "configure Y" without specifics)
- [ ] Dependencies between tasks are named
- [ ] File paths are specific (not "create auth module" but "create `src/auth/mod.rs`")
- [ ] Sub-skill assigned to each task

## ADR Google OAuth для MLMaiL: loopback + Credential Manager + platform storage

**Контекст:** MLMaiL потребує Google-авторизації з доступом до `gmail.modify` scope на двох платформах — macOS (Tauri desktop) і Android — без єдиного рішення у вигляді сторонньої бібліотеки, що покриває обидві.

**Рішення/Процедура/Факт:**

- macOS: loopback HTTP server (127.0.0.1:RANDOM_PORT) через `tokio::net::TcpListener` + OAuth 2.0 Authorization Code + PKCE (`base64`/`sha2`/`rand`), redirect в системний браузер через `tauri-plugin-opener`, обмін code → tokens через `reqwest` (rustls-tls), refresh token → Keychain через `keyring` crate (apple-native feature).
- Android: Credential Manager bottom-sheet (sign-in stage) → Identity Authorization API з `requestOfflineAccess(webClientId, forceCodeForRefreshToken=true)` (server auth code stage) → JNI-міст у Kotlin-модуль `MlmailAuth`, далі той самий `token_exchange.rs` у Rust обмінює server auth code на tokens; refresh token → `EncryptedSharedPreferences` через `SecureStore.kt` (master key у Android Keystore).
- Rust — єдиний holder токенів: `AuthState` in-memory (access token + expiry), storage — platform (Keychain / EncryptedSharedPreferences). Frontend (`app/src/services/auth-store.js`) отримує access token через Tauri-команду `auth_get_access_token`, не зберігає токенів у JS.
- PKCE використовується лише на macOS (Desktop OAuth client); на Android — немає (код підписаний SHA-1 fingerprint APK).
- Помилки серіалізуються як `{kind, message}`, Vue показує українські тексти через `app/src/i18n/auth-errors.js`.
- Нові файли: `app/src-tauri/src/auth/{mod,state,pkce,id_token,token_exchange,config}.rs`, `app/src-tauri/src/auth/storage/{mod,macos,android}.rs`, `app/src-tauri/src/auth/flow/{macos,android}.rs`; Kotlin: `app/src-tauri/gen/android/app/src/main/java/com/vitaliytv/mlmail/auth/`; Vue: `app/src/views/Login.vue`, `app/src/services/auth-store.js`, `app/src/i18n/auth-errors.js`.
- Spec: `docs/superpowers/specs/2026-05-11-google-oauth-design.md`; план (10 задач, 3 фази): `docs/superpowers/plans/2026-05-11-google-oauth.md`.

**Обґрунтування:** Platform-native підхід не вимагає custom URI scheme (не потрібен deep link на macOS), loopback дозволяє Desktop OAuth client type без реєстрації фіксованого redirect URI. Credential Manager — Google-recommended шлях для sign-in на Android; Authorization API — єдиний підтримуваний спосіб отримати `serverAuthCode` з необхідними scopes для offline access. Rust-тільки токен-сховище усуває XSS-ризик витоку access token через JS. Rustls замість OpenSSL спрощує Android NDK build chain.

**Розглянуті альтернативи:**

- AppAuth-Android (OpenID Foundation) — відкинуто на користь Credential Manager (офіційна Google-рекомендація для нових застосунків).
- GoogleSignIn legacy API — deprecated, відкинуто.
- `tauri-plugin-oauth` community plugin (замість власного loopback) — відкинуто: додатковий непідконтрольний шар.
- Stronghold / Tauri encrypted store для refresh token — відкинуто: зайвий рівень замість platform-native KMS.
- Зберігати токени у Vue state / localStorage — відкинуто через XSS-ризик.
