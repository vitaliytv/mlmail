## ADR Архітектура Google OAuth-авторизації у MLMaiL (macOS + Android)

**Контекст:** MLMaiL — Tauri 2 + Vue 3 застосунок, якому потрібен доступ до Gmail API. Стандартний implicit/SPA flow неприйнятний для нативних клієнтів: refresh_token у JavaScript — пряма атака surface.
**Рішення/Процедура/Факт:** Реалізовано Authorization Code flow у двох варіантах за платформою: (1) **macOS** — PKCE + loopback HTTP listener (`tokio::net::TcpListener`) на випадковому порту, `tauri-plugin-opener` відкриває браузер, callback повертає code, Rust обмінює на `access_token` + `refresh_token` через `reqwest → oauth2.googleapis.com/token`; Keychain через `keyring` crate (`apple-native` feature); (2) **Android** — Credential Manager (`GetGoogleIdOption`) для sign-in → `GoogleIdTokenCredential`, плюс `AuthorizationClient` (`Identity.getAuthorizationClient`) з `requestOfflineAccess(forceCodeForRefreshToken=true)` для `serverAuthCode`, Kotlin-модуль як Tauri 2 mobile plugin (`@TauriPlugin`), EncryptedSharedPreferences з AES256_GCM через Android Keystore. Rust тримає всі токени; Vue отримує лише `access_token` через Tauri-команду `auth_get_access_token` і ніколи не бачить `refresh_token`. Нові файли: `app/src-tauri/src/auth/{pkce,id_token,state,error,config,token_exchange}.rs`, `auth/flow/{macos,android}.rs`, `auth/storage/{macos,android,in_memory}.rs`, `app/src/views/Login.vue`, `app/src/services/auth-store.js`, `app/src/i18n/auth-errors.js`, Kotlin `auth/{MlmailAuthPlugin,CredentialManagerFlow,AuthorizationFlow,SecureStore}.kt`, `docs/adr/ADR-0006-google-oauth.md`.
**Обґрунтування:** Таури-команди як єдиний IPC bridge ізолюють токени від WebView (XSS не дістанеться `refresh_token`). `keyring` з `apple-native` прибирає D-Bus залежність на macOS. `AuthorizationClient.requestOfflineAccess(forceCodeForRefreshToken=true)` — єдиний спосіб отримати `serverAuthCode` від Android Credential Manager без deprecated `GoogleSignIn`. PKCE на desktop дозволяє Desktop-type OAuth client без `client_secret`.
**Розглянуті альтернативи:** AppAuth-Android (окрема бібліотека, зайва складність, deprecated); `tauri-stronghold` для storage (зайвий рівень поверх системного Keychain/Keystore); зберігати токен у Vue store (відкидає захист від XSS); implicit flow (немає `refresh_token`); серверний проксі для token exchange (нема бекенду).
**Зачіпає:** `app/src-tauri/src/auth/*`, `app/src-tauri/gen/android/.../auth/*`, `app/src/views/Login.vue`, `app/src/services/auth-store.js`, `app/src/i18n/auth-errors.js`, `app/src-tauri/Cargo.toml`, `app/src-tauri/gen/android/app/build.gradle.kts`, `docs/ci4/02-containers.md`, `docs/ci4/03-components.md`, `docs/ci4/04-code.md`, `docs/ci4/decisions.md`.

## Knowledge `setServerClientId` у Credential Manager вимагає Web OAuth client ID, не Android

**Контекст:** При інтеграції Android Credential Manager (`GetGoogleIdOption`) виникає неочевидна деталь щодо типу OAuth client ID, який передається як `serverClientId`.
**Рішення/Процедура/Факт:** `GetGoogleIdOption.Builder().setServerClientId(...)` вимагає **Web application** client ID з Google Cloud Console — не Android client ID. Сам APK ідентифікується через **Android** OAuth client (package name + SHA-1 fingerprint), але `serverClientId` — завжди Web. Отже у проєкті потрібні два окремі client ID для Android-flow: один Android-тип і один Web-тип.
**Обґрунтування:** Це задокументована особливість Google Credential Manager API — `serverClientId` вказує, від імені якого серверного клієнта буде отримано ID token і server auth code, тому Google вимагає Web-тип client.
**Розглянуті альтернативи:** не обговорювалися.
**Зачіпає:** `app/src-tauri/src/auth/config.rs` (константи `ANDROID_CLIENT_ID` і `ANDROID_WEB_CLIENT_ID`), `app/src-tauri/.env.example`, Kotlin `CredentialManagerFlow.kt`.

## Knowledge OAuth Client IDs не створюються через `gcloud` CLI

**Контекст:** Розробник запитав, як створити три OAuth client IDs у Google Cloud через термінал для project `vitaliytv`.
**Рішення/Процедура/Факт:** OAuth 2.0 Client IDs у Google Cloud Console **неможливо створити через `gcloud` CLI** — це обмеження Google API. OAuth consent screen External-типу теж лише через UI. Через термінал можна: `gcloud services enable gmail.googleapis.com`, `keytool` для debug SHA-1, і редагування `.env`. Самі client IDs — тільки через `https://console.cloud.google.com/apis/credentials?project=<project_id>`.
**Обґрунтування:** Google не надає CLI або REST endpoint для створення OAuth clients — тільки через Cloud Console UI.
**Розглянуті альтернативи:** не обговорювалися.
**Зачіпає:** `app/src-tauri/.env.example`, процес onboarding нового розробника.
