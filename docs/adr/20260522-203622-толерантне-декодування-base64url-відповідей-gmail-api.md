## ADR Толерантне декодування base64url відповідей Gmail API

## Context and Problem Statement

`body.data` у відповідях Gmail API містить стандартний base64url із `=`-паддингом. Попереднє рішення використовувало двигун `URL_SAFE_NO_PAD`, який відхиляє паддинг — тіло листів, довжина яких не кратна 3 байтам, завжди поверталось порожнім.

## Considered Options

- Кастомний `base64::engine::GeneralPurpose` з `DecodePaddingMode::Indifferent`
- Інші варіанти в transcript не обговорювалися.

## Decision Outcome

Chosen option: "Кастомний engine з `DecodePaddingMode::Indifferent`", because тимчасовий `eprintln!`-дамп MIME-дерева підтвердив `b64=ERR(Invalid padding)` для обох частин (`text/plain`, `text/html`) конкретного листа; двигун без паддингу приймає обидва формати, що усуває помилку для всіх листів незалежно від вирівнювання довжини тіла.

### Consequences

- Good, because transcript фіксує очікувану користь: 14/14 unit-тести модуля `gmail::message` проходять, зокрема новий тест `extract_plain_text_decodes_padded_base64url`, який раніше давав `left: ""`.
- Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

- Файл: `app/src-tauri/src/gmail/message.rs` — константа `BASE64URL`, функція `decode_part()`.
- Залежність: `base64 = "0.22"` (вже існувала), `base64::engine::DecodePaddingMode`, `base64::alphabet::URL_SAFE`.
- Тест: `gmail::message::tests::extract_plain_text_decodes_padded_base64url` — кодував `"aGVsbG8="` через `URL_SAFE_NO_PAD`, очікував `"hello"`, спочатку падав (RED), після фіксу — зелений.
- Попередній двигун: `base64::engine::general_purpose::URL_SAFE_NO_PAD`.

## ADR Декодування нон-UTF-8 charset у MIME-частинах через `encoding_rs`

## Context and Problem Statement

Gmail API повертає MIME-частини з різними кодуваннями (`ISO-8859-1`, `windows-1251`, `gb2312`, `shift_jis` тощо). Попередня функція `decode_base64url` викликала `String::from_utf8(bytes).ok()?`, яка повертає `None` для будь-якого байта ≥ 0x80 у нон-UTF-8 charset — тіло таких листів завжди було порожнім, причому параметр `charset` із `Content-Type` код взагалі не читав.

## Considered Options

- `encoding_rs` — крейт Mozilla для перекодування Web-кодувань
- Інші варіанти з питання в transcript частково обрізані й не відтворюються без домислів.

## Decision Outcome

Chosen option: "`encoding_rs`", because користувач явно обрав цей варіант як рекомендований після того, як transcript підтвердив `charset=ISO-8859-1` на реальному листі з порожнім тілом; крейт охоплює всі Web-кодування (включно з windows-1251, gb2312, shift_jis), декодування lossy — непорожні дані завжди дають текст.

### Consequences

- Good, because transcript фіксує очікувану користь: нові тести `extract_plain_text_decodes_iso_8859_1_body` та `extract_plain_text_decodes_windows_1251_body` спершу падали (RED), після реалізації — зелені; 93/93 тестів проходять.
- Bad, because transcript не містить підтверджених негативних наслідків.

## More Information

- Файл: `app/src-tauri/Cargo.toml` — `encoding_rs = "0.8"`.
- Файл: `app/src-tauri/src/gmail/message.rs` — функція `part_charset()` читає параметр `charset` із `Content-Type`; `decode_part()` передає байти через `encoding_rs::Encoding::for_label()`, fallback — UTF-8 lossy.
- Тести: `gmail::message::tests::extract_plain_text_decodes_iso_8859_1_body`, `..._windows_1251_body`.
- Версія пакета: `app` `0.1.2` → `0.1.3`; запис у `app/CHANGELOG.md` розділ `[0.1.3] - 2026-05-22`.
