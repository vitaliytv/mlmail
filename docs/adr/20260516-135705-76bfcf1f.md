## ADR Показ випадкового листа INBOX: вибір sample space і витяг тексту

**Контекст:** Після логіну стартовий екран MLMaiL показує один випадковий лист із INBOX, що дає живий preview акаунта. Необхідно обрати: (а) звідки брати «випадковий» id, (б) як отримати читаємий текст без HTML-рендерингу.

**Рішення/Процедура/Факт:**

- `gmail_random_message` (Rust, `app/src-tauri/src/gmail/mod.rs`) робить два HTTP-виклики: спочатку `GET messages?labelIds=INBOX&maxResults=100&fields=messages/id` → масив id, потім `GET messages/<rand_id>?format=full` → парсинг.
- Sample space — останні 100 листів; `rand::random` index всередині масиву.
- Витяг тіла (`app/src-tauri/src/gmail/message.rs`, `extract_plain_text`): рекурсивний обхід `payload.parts`, перевага `text/plain` над `text/html`; HTML-fallback — `regex` strip тегів + `html-escape` для decoding entities; truncate до 10 000 символів.
- `GmailError::Empty` — новий варіант для порожнього INBOX.
- Vue: `auth-store.js` отримав `currentMessage`, `messageErrorKind`, `isMessageLoading`, `loadRandomMessage()`; `Login.vue` — картка From/Тема/Дата + `<pre>` тіла + кнопка «Показати інший»; i18n `auth-errors.js` доповнений ключем `Empty`.

**Обґрунтування:** `messages.list?maxResults=100` + `messages.get` — два виклики з детермінованою швидкістю без пагінації. Альтернатива «чесна випадковість по всьому INBOX» потребувала б десятків послідовних сторінкових запитів (50+ для 5K листів), що неприйнятно для UX. HTML-рендер виключено на цій ітерації через складність sanitization у Tauri WebView.

**Розглянуті альтернативи:**

- Рандом по всьому INBOX через pageToken-пагінацію — відхилено через непередбачуваний час відповіді.
- Рандом по даті (query `before:/after:`) — відхилено через складний розрахунок і rounding-ефекти.
- Повний HTML-рендер через DOMPurify / `<iframe sandbox>` — відкладено на наступну ітерацію.

**Зачіпає:** `app/src-tauri/src/gmail/mod.rs`, `app/src-tauri/src/gmail/message.rs`, `app/src-tauri/src/gmail/error.rs`, `app/src-tauri/Cargo.toml` (deps: `regex`, `html-escape`), `app/src/services/auth-store.js`, `app/src/views/Login.vue`, `app/src/i18n/auth-errors.js`, `docs/ci4/03-components.md`, `docs/ci4/04-code.md`, `docs/adr/_inbox/20260515-221525-random-message.md`

## Knowledge Quasar vs Reka UI vs plain HTML для Vue-компонентів у MLMaiL

**Контекст:** Cursor-правило `n-vue.mdc` вказує «Використовуй Quasar для компонентів», але Quasar у проєкті не встановлений; `Login.vue` наразі на plain HTML зі scoped CSS.

**Рішення/Процедура/Факт:** Порівняння трьох підходів (сесія 2026-05-16):

- **Plain HTML** — 0 KB overhead, повний контроль CSS, ARIA пишеться вручну; підходить для мінімального MVP.
- **Quasar** — ~120 KB gzip runtime, готові `q-btn` (loading, ripple, іконки), `q-card`, `q-banner`, `$q.notify`/`$q.dialog` з a11y; Material-look out-of-the-box; вимагає `@quasar/vite-plugin` + `Quasar` plugin у `main.js` + sass-vars файл; відповідає правилу `n-vue.mdc`.
- **Reka UI** — ~10–20 KB tree-shaken headless-примітиви (діалоги, tooltip, combobox з a11y/keyboard); жодного дизайн-вибору не нав'язано, але спінери/картки/банери — голий HTML + власний CSS; найкраще парується з Tailwind або shadcn-vue.

**Обґрунтування:** Рішення про встановлення Quasar (або зміну правила) ще не прийнято — сесія завершилась на стадії порівняння. Якщо UI зростатиме до десятка діалогів/таблиць, Quasar економить час; якщо планується власна design-система на Tailwind — Reka UI.

**Розглянуті альтернативи:** не обговорювалися поза трьома вище.

**Зачіпає:** `app/src/views/Login.vue`, `app/vite.config.js`, `app/src/main.js`, `.cursor/rules/n-vue.mdc`
