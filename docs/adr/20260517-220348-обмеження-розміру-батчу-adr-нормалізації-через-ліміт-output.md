## ADR Обмеження розміру батчу ADR-нормалізації через ліміт output-токенів

## Context and Problem Statement
Автоматичний хук `.claude/hooks/normalize-decisions.sh` запускається з дефолтним батчем 30 чернеток ADR за раз. Під час прогону з `ADR_NORMALIZE_BATCH=30` (default) відповідь `claude sonnet` перевищила 32k output-токенів і скрипт аварійно завершився з помилкою до запису будь-яких файлів.

## Considered Options
* `ADR_NORMALIZE_BATCH=30` (default) — один прогон охоплює всі чернетки, але перевищує output-ліміт Sonnet
* `ADR_NORMALIZE_BATCH=10` — рекомендований скілом розмір, dry-run підтвердив коректний вихід
* `ADR_NORMALIZE_BATCH=5` — обраний після таймауту BATCH=10 на реальному прогоні

## Decision Outcome
Chosen option: `ADR_NORMALIZE_BATCH=5` (зі зниженням через `ADR_NORMALIZE_BATCH=10`), because dry-run з BATCH=10 пройшов успішно, але реальний прогон із BATCH=10 завис на ~1 годину й упав через `Request timed out`; користувач обрав BATCH=5 як більш надійний розмір для реального виконання.

### Consequences
* Good, because transcript фіксує очікувану користь: dry-run з BATCH=10 завершився без помилок і видав коректний план (2 merge, 4 delete, 4 rewrite).
* Bad, because Neutral, because transcript не містить підтвердження наслідку щодо загальної кількості прогонів, необхідних для обробки всіх ~33 чернеток при BATCH=5.

## More Information
- Скрипт: `.claude/hooks/normalize-decisions.sh`
- Лог: `.claude/hooks/normalize-decisions.log`
- Змінні керування: `ADR_NORMALIZE_THRESHOLD=0`, `ADR_NORMALIZE_MIN_INTERVAL_HOURS=0`, `ADR_NORMALIZE_BATCH=5`, `ADR_NORMALIZE_DRY=1`
- Модель: `claude sonnet` (виклик через `claude -p`)
- Dry-run з BATCH=10 виконався за ~10 хвилин; реальний прогон BATCH=10 завис понад 1 годину до таймауту
- Guardrail від рекурсії: паралельні Stop-хук-виклики логували `skip: only N s since last attempt` — основний процес не переривався
