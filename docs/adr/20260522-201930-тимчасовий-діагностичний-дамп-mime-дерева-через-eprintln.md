## ADR Тимчасовий діагностичний дамп MIME-дерева через `eprintln!`

## Context and Problem Statement

У деяких листів тіло повертається порожнім. Виявлено три гіпотетичні причини в `app/src-tauri/src/gmail/message.rs`: (1) порожній `text/plain`-блок блокує HTML-фолбек, (2) лист зі самих зображень після `strip_html` не містить тексту, (3) не-UTF-8 charset мовчки дає `None`. Перед виправленням потрібно зібрати докази для конкретних листів.

## Considered Options

- Тимчасові `log::debug!` / `log::info!` через crate `log`
- Тимчасові `eprintln!` у stderr

## Decision Outcome

Chosen option: "`eprintln!` у stderr", because у `app/src-tauri/src/lib.rs` не ініціалізовано жодного backend'у для crate `log` — макроси `log::*` були б беззвучні; `eprintln!` гарантовано виводиться у термінал `tauri dev`.

### Consequences

- Good, because transcript фіксує очікувану користь: одна функція `debug_describe_payload()` рекурсивно обходить MIME-дерево і виводить `mimeType`, розмір, наявність `body.data`, результат base64/UTF-8-декодування, 120-символьний preview і `Content-Type` з charset — це підтверджує всі три гіпотези за один прогон без повторних перебудов.
- Bad, because інструментацію позначено `// TEMP DEBUG (empty-body diagnosis) — remove after root cause confirmed` і треба вручну прибрати після підтвердження причини.

## More Information

Змінені файли:

- `app/src-tauri/src/gmail/message.rs` — нова функція `debug_describe_payload()`, позначена TEMP DEBUG
- `app/src-tauri/src/gmail/mod.rs` — тригер у `get_message_at`: якщо `body_text.trim().is_empty()`, викликається `debug_describe_payload(payload)` через `eprintln!("[mlmail][TEMP-DEBUG] ...")`

Перевірено: `cargo check --lib` і `cargo clippy --lib` — без нових помилок чи попереджень. Єдине попередження clippy — у pre-existing коді в `auth`.
