## ADR Команда `gmail_random_message`: вибірка з 100 листів і MIME-парсинг

**Контекст:** MLMaiL потребує показувати один випадковий лист INBOX після логіну, щоб одразу дати користувачу відчуття «живого» поштового клієнта. Потрібно обрати простір вибірки і спосіб отримати читабельне тіло листа з API Gmail.

**Рішення/Процедура/Факт:** Реалізовано два послідовних HTTP-виклики: `GET messages?labelIds=INBOX&maxResults=100&fields=messages/id` → псевдовипадковий `rand::thread_rng().gen_range` індекс → `GET messages/<id>?format=full`. Для тіла рекурсивно обходиться `payload.parts` (і `payload` на верхньому рівні): першим береться `text/plain` частина (base64url-decode), за відсутності — `text/html` зі стрипом тегів через `regex` + декодуванням HTML-ентитей через `html-escape`. Тіло обрізається до 10 000 символів. Нові файли: `app/src-tauri/src/gmail/message.rs` (DTO `GmailMessage { id, from, subject, date, body }` + хелпери `extract_header`, `extract_plain_text`). Tauri-команда `gmail_random_message` додана у `lib.rs`. На Vue-боці `auth-store.js` отримав `currentMessage` / `messageErrorKind` / `isMessageLoading` і метод `loadRandomMessage()`, який викликається авто після `initialize()`/`login()` і вручну через кнопку «Показати інший». `GmailError::Empty` доданий для порожнього INBOX. i18n-ключ `Empty` → «Скринька порожня.»

**Обґрунтування:** Вибірка з 100 останніх — два HTTP-виклики, стабільна квота API, достатня «випадковість» для UX. Повний INBOX-обхід через pageToken-пагінацію дасть справедливу випадковість, але для скриньки в 5K+ листів — це 50+ послідовних запитів. Пріоритет `text/plain` над `text/html` — простіше, безпечно (без XSS-ризику), не потребує DOMPurify. HTML-стрип через `regex` є достатнім для fallback: він не є парсером HTML, але для показу контенту листа в `<pre>` цього вистачає.

**Розглянуті альтернативи:** Рандом по всьому INBOX через pageToken (відкинуто через квоту), `q=before:/after:` Gmail query (складно, округлення по даті), повний HTML-рендер через DOMPurify+iframe (занадто великий scope для першої ітерації).

**Зачіпає:** `app/src-tauri/src/gmail/message.rs` (новий), `app/src-tauri/src/gmail/mod.rs`, `app/src-tauri/src/gmail/error.rs`, `app/src-tauri/src/lib.rs`, `app/src/services/auth-store.js`, `app/src/views/Login.vue`, `app/src/i18n/auth-errors.js`, `app/src-tauri/Cargo.toml` (додано `regex = "1"` і `html-escape = "0.2"`).

## Knowledge `promise/avoid-new` oxlint: workaround для deferred promise у тестах

**Контекст:** oxlint-правило `promise(avoid-new)` забороняє `new Promise(...)` навіть у тестовому коді, де deferred-паттерн є єдиним простим способом відкласти resolve до потрібного моменту тесту (`resolveCount` ззовні closure).

**Рішення/Процедура/Факт:** Перед `new Promise(...)` у тест-файлі додається коментар `// oxlint-disable-next-line promise/avoid-new`. eslint-disable не допомагає (правило живе в oxlint). Водночас сам тест переписаний так, що deferred promise потрібен лише якщо логіка справді sequential; якщо можна — краще `vi.fn().mockResolvedValueOnce(...)`.

**Обґрунтування:** Правило справедливе для продакшн-коду, але у тестах deferred promise — законний паттерн для перевірки intermediate loading-стейтів. Disable-коментар на один рядок — мінімальний виняток без відключення правила глобально.

**Розглянуті альтернативи:** не обговорювалися.

**Зачіпає:** `app/src/views/Login.test.js:98` (рядок з `// oxlint-disable-next-line`).

## ADR Вибір UI-бібліотеки для Tauri macOS + Android: Quasar

**Контекст:** Правило `.cursor/rules/vue.mdc` вимагає використовувати Quasar для Vue-компонентів MLMaiL. У сесії проведено порівняння трьох варіантів (plain HTML, Quasar, Reka UI) з урахуванням платформ macOS і Android.

**Рішення/Процедура/Факт:** Рекомендовано Quasar; рішення ще не зафіксовано у коді — наступна ітерація. Поточний `Login.vue` використовує plain HTML. Рішення відкладено до окремого spec→plan→execute циклу.

**Обґрунтування:** Для Tauri-застосунку на macOS+Android Quasar дає Material Design UX, який є нативним стандартом на Android; safe-area, swipe-to-dismiss, `$q.notify`/`$q.dialog` — компоненти, які точно знадобляться у наступних ітераціях (список листів, перегляд, AI-саммері). Bundle +120 КБ gzip не критичний у рамках Tauri-бінарника. Reka UI (headless) вимагав би писати весь CSS вручну. Plain HTML — порушення існуючого правила.

**Розглянуті альтернативи:** Plain HTML (нуль deps, але порушує правило; немає mobile UX компонентів), Reka UI (~10–20 КБ tree-shaken, повний контроль над CSS, a11y з коробки — але немає готових banner/spinner/card).

**Зачіпає:** `app/src/views/Login.vue`, `app/src/main.js`, `app/vite.config.js`, `app/package.json` — зміни будуть у наступному ітерейшні.
