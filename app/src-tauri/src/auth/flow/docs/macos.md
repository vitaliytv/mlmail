## Огляд

`run_login_flow` запускає desktop OAuth-вхід у Google через браузер, спрямовуючи користувача на <https://accounts.google.com/o/oauth2/v2/auth> і очікуючи локальне повернення на loopback-адресу `http://127.0.0.1:{port}/callback`; у тестовому сценарії також приймає формат `http://127.0.0.1:7777/cb`. Це потрібно, щоб застосунок міг отримати авторизаційний результат для доступів <https://www.googleapis.com/auth/gmail.modify> і <https://www.googleapis.com/auth/gmail.settings.basic> без ручного копіювання даних із браузера.

Функція звертається до мережі, але працює в режимі read-only: не пише у ФС або БД. Збої під час відкриття браузера, локального callback чи OAuth-обміну вона перехоплює всередині й не викидає винятки назовні; за частини помилкових сценаріїв повертає порожнє значення, наприклад `null`.

## Поведінка

1. `run_login_flow` запускає desktop OAuth-вхід через браузер, щоб отримати Google-токени для роботи з Gmail без ручного копіювання коду користувачем.

2. Перед стартом відкриває авторизацію на `https://accounts.google.com/o/oauth2/v2/auth` і формує локальну адресу повернення `http://127.0.0.1:{port}/callback`, щоб Google міг повернути результат у застосунок на тимчасовий loopback-порт.

3. Запитує доступи `https://www.googleapis.com/auth/gmail.modify` і `https://www.googleapis.com/auth/gmail.settings.basic` разом з базовою ідентифікацією користувача, щоб застосунок міг працювати з листами та базовими Gmail-налаштуваннями.

4. Використовує захист від підміни OAuth-відповіді та після повернення з браузера звіряє, що відповідь належить саме поточній спробі входу.

5. Очікує одноразовий callback із браузера протягом обмеженого часу; якщо користувач не завершує вхід вчасно, завершує сценарій контрольованою помилкою без винятків назовні.

6. Після отримання callback одразу показує у браузері коротке підтвердження, що вхід завершено і вкладку можна закрити, щоб користувач мав явний візуальний сигнал про успіх локального повернення.

7. Приймає з callback код авторизації або помилку від Google; якщо Google повернув відмову чи callback неповний або некоректний, повертає контрольовану помилку.

8. Обмінює отриманий код на токени desktop-flow і повертає результат далі в застосунок для подальшої автентифікованої роботи.

9. Працює тільки в межах read-only щодо ФС/БД: нічого не записує у файлову систему чи базу даних, але звертається до мережі та до локального HTTP callback.

10. Не кидає винятків назовні: збої відкриття браузера, мережеві проблеми, помилки callback і помилки OAuth перетворює на керований результат із помилкою.

11. У тестовому сценарії підтримується типова адреса повернення на кшталт `http://127.0.0.1:7777/cb`, що фіксує очікуваний формат loopback redirect для desktop OAuth.

## Гарантії поведінки

- Read-only: не виконує операцій запису (ФС/БД).
- Перехоплює помилки і не пропускає винятків назовні (fail-safe).
- За певних помилок повертає порожнє значення (напр. `null`) замість винятку.
