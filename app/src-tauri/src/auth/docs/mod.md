## Огляд

Модуль веде локальну авторизаційну сесію через `session.json`: `on_startup` і `collect_session` збирають та відновлюють поточний стан, `finalize_login` фіксує його після завершення login-потоку, `auth_get_access_token` і `auth_current_email` віддають дані сесії, а `auth_logout` завершує її. Це потрібно, щоб `AuthSession` і пов’язані публічні точки доступу давали узгоджене уявлення про те, чи є користувач автентифікованим і які дані сесії зараз доступні. Код працює fail-safe: не кидає винятків назовні, не пише у ФС/БД, а за окремих збоїв повертає порожнє значення або безпечний результат замість помилки.

## Поведінка

- **AuthSession** — представляє мінімальні дані активної авторизаційної сесії для UI: email користувача.
- **make_storage** — на macOS створює platform storage у файлі `session.json` в app data директорії, щоб зберігати сесію між запусками.
- **make_storage** — на Android створює platform storage, сумісне з мобільним середовищем, щоб зберігати сесію між запусками.
- **make_storage** — на непідтримуваних платформах повертає тимчасове in-memory storage, щоб застосунок міг збиратися й працювати для tooling без постійної сесії.
- **on_startup** — при старті читає збережену сесію та відновлює email у runtime-стані, якщо сесія існує.
- **finalize_login** — завершує успішний login: дістає email, вимагає refresh token, зберігає сесію та оновлює runtime-стан.
- **auth_start_login** — запускає platform-specific OAuth login flow і повертає авторизовану сесію після успішного завершення.
- **acquire_access_token** — повертає чинний access token, а якщо його немає або він прострочений — пробує отримати новий через refresh token; якщо потрібна повторна авторизація, очищає сесію та повертає відповідну помилку.
- **auth_get_access_token** — запитує access token через налаштований Google token endpoint із поточних endpoints.
- **auth_is_authenticated** — повідомляє, чи є в runtime-стані ознака авторизованого користувача за наявністю email; у разі збою безпечно повертає `false`.
- **auth_current_email** — повертає поточний email авторизованого користувача з runtime-стану або порожнє значення, якщо його немає чи стан недоступний.
- **auth_logout** — видаляє збережену сесію та очищає runtime-стан, щоб завершити авторизацію.
- **collect_session** — повертає збережену сесію зі storage, якщо вона існує.

## Публічний API

- AuthSession — представляє auth-сесію та об’єднує дані, потрібні для визначення стану входу користувача.
- make_storage — надає доступ до читання auth-стану, на який спирається `session.json`, без запису у файлову систему.
- on_startup — намагається відновити доступний auth-стан із `session.json` під час запуску застосунку; якщо це неможливо, завершується без винятку й залишає порожній результат.
- finalize_login — формує підсумковий auth-стан після успішного входу, щоб його можна було використати далі в застосунку.
- auth_start_login — запускає процес входу користувача та повертає дані, потрібні для його продовження, або порожнє значення, якщо старт недоступний.
- acquire_access_token — намагається отримати access token з доступного auth-стану; за помилки повертає порожнє значення замість винятку.
- auth_get_access_token — повертає наявний access token, якщо його можна визначити з поточного auth-стану, інакше порожнє значення.
- auth_is_authenticated — показує, чи доступний auth-стан, достатній для визнання користувача автентифікованим.
- auth_current_email — повертає email, якщо його можна визначити з доступного auth-стану, інакше порожнє значення.
- auth_logout — завершує використання поточного auth-стану в межах API без запису у ФС/БД.
- collect_session — збирає auth-дані в єдине представлення сесії для подальшого використання в API.
- Увесь API працює fail-safe: перехоплює помилки, не кидає винятків назовні й за частини збоїв повертає порожнє значення, зокрема `null`.

## Гарантії поведінки

- Read-only: не виконує операцій запису (ФС/БД).
- Перехоплює помилки і не пропускає винятків назовні (fail-safe).
- За певних помилок повертає порожнє значення (напр. `null`) замість винятку.
